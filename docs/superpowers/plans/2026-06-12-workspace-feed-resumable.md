# Workspace and State: Revision Feed and Memory-Snapshot Pairing (W4, slice 4, final)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Complete W4 with the two remaining pieces: (1) the REVISION CHANGE FEED, so external indexers learn about workspace activity without polling: the controller emits CloudEvents (`dev.mitos.workspace.revision.created`, `dev.mitos.sandbox.phase.changed`) to a configured sink (an HTTP webhook; NATS is a later option), AND mirrors each as a Kubernetes Event on the object, with no embedded vector DB; and (2) MEMORY-SNAPSHOT PAIRING, so a workspace head can be RESUMABLE: when a workspace-bound sandbox is checkpointed (its VM memory snapshotted alongside the dehydrate), the resulting WorkspaceRevision is paired with that memory snapshot (`memorySnapshotRef` set), `status.resumable` becomes true, and resuming a resumable head restores the VM memory AND the workspace content together. Verified: the feed emits a CloudEvent to a fake sink + a Kubernetes Event on revision creation and on sandbox phase change (envtest + unit); the pairing sets `memorySnapshotRef` when a memory snapshot accompanies the dehydrate and `status.resumable` reflects it, and a resume of a resumable head requests both the memory snapshot restore and the workspace hydrate (envtest with seams); the real VM-memory restore is the KVM/bare-metal tail (the husk #18 boundary), documented.

**Honesty constraint:** the CloudEvents feed is delivered to whatever sink is configured (an HTTP webhook URL); we prove it against a fake/local sink, not an external indexer, and the docs say the feed CONTRACT (event types + payload schema) is the deliverable, with the reference turbovec indexer out of scope. The Kubernetes Event mirroring is the on-cluster, always-available channel; the CloudEvents sink is opt-in (off by default; no sink = Events only). The memory-snapshot pairing's MODEL + status are envtest-proven; the actual VM-memory restore needs the VMM and is proven on KVM or documented as the bare-metal tail (do not claim a memory resume that a KVM job does not show). "resumable" is honest: status.resumable is true only when the head revision actually carries a memorySnapshotRef whose snapshot exists; a paired snapshot that was GC'd flips resumable false. The memory snapshot is principal-bound per the secrets policy (a memory snapshot can contain secrets in RAM; it is paired to the workspace's grant principal and never served cross-principal); state this. At-least-once feed delivery with an idempotency key (the event carries the revision UID + a monotonic id) so an indexer can dedupe; we do not promise exactly-once.

**Architecture:** `internal/eventfeed` (new): a CloudEvents emitter with a sink interface (an HTTP-webhook sink; a no-op sink when unconfigured) that builds a CloudEvent (spec-version, type, source, subject, id, time-in-from-args-not-Now, data) and POSTs it; the controller calls it on WorkspaceRevision creation and on SandboxClaim phase transitions, and also records a Kubernetes Event via the manager's EventRecorder. The feed sink URL + on/off is a controller flag. For the pairing: the dehydrate-on-terminate path (slice 2/3), when the run path also produced a VM memory snapshot (the husk Checkpoint primitive / a checkpoint-on-terminate option), sets `WorkspaceRevision.spec.memorySnapshotRef` to the snapshot ref; the Workspace controller (slice 1) already computes `status.resumable` from the head's memorySnapshotRef, extended to verify the snapshot still exists; the hydrate-on-activate path, for a resumable head, requests the memory-snapshot restore (the husk activation restores that memory image instead of the template's) plus the workspace hydrate, paired and principal-checked.
