# Encryption Key Custody Implementation Plan (issue #31, PR2)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the PR2 hardening on issue #31. Move the encryption key off the node: the controller owns each per-template (per-scope) key in a Kubernetes Secret and delivers it to forkd over the existing mTLS gRPC channel; forkd holds the key only in memory and NEVER writes it to the node data disk. Crypto-shred becomes controller-driven: when a template is torn down, the controller deletes the key Secret and forkd wipes the LUKS keyslots and zeroizes the in-memory key, so the ciphertext is unrecoverable and the key exists nowhere. This replaces PR1's `InMemoryKeyProvider` (which generated the key on the node) with a key whose authoritative custody is etcd (the Secret), distributed per scope, never persisted to the forkd data disk. The encryption mechanism itself (per-scope dm-crypt/LUKS containers, CoW-preserved, crypto-shred via luksErase) already shipped in PR1; this PR changes WHERE the key lives and WHO controls its lifecycle.

**Honesty constraint:** the key is a secret value: it travels only over the mTLS gRPC channel (encrypted in transit), is never logged, never in an error message, never in a span/audit record, and is stored at rest only in the Kubernetes Secret (whose at-rest protection is the cluster's etcd-encryption responsibility, which we state as an assumption, not a claim we provide). forkd holds the key in process memory for the lifetime of an open container and zeroizes it on close/shred; it is never written to `<dataDir>` in any form. We document precisely the trust boundary (the controller and etcd are trusted with the key; the node disk is not) and what is verified (envtest proves the Secret lifecycle and that the key flows over the RPC; a check proves forkd never writes the key to the data disk; the storecrypt KVM CI from PR1 proves the mechanism). The threat-model row is updated.

**Architecture:** The controller generates a 256-bit key per template scope and stores it in a Secret (`<template>-enc-key`, type Opaque, key `key`) in the operator namespace, created when a pool/template with encryption enabled is reconciled. The key is delivered to forkd in the `CreateTemplate` RPC (a new `encryption_key` bytes field) and in the `Fork` RPC (so forkd can reopen a closed container after a restart without the controller having to re-push out of band). forkd swaps `InMemoryKeyProvider` for a `RequestKeyProvider`: the key for a scope is the one carried in the current request; it is cached in memory while the container is open and zeroized on forget/shred; it is never persisted. On template teardown the controller deletes the Secret and invokes forkd to shred the container (via the existing/added DeleteTemplate-or-Shred path); the in-memory key and the LUKS keyslots are destroyed.
