# Snapshot Version-Compatibility Contract Implementation Plan (issue #32)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close issue #32, a format-freeze blocker. Make the snapshot a versioned artifact with an enforced compatibility contract so the snapshot format can be safely frozen for production tenants. Firecracker does not guarantee that a snapshot taken by one VMM version (or on one CPU model) restores correctly under another; restoring across an incompatible boundary can crash the guest or silently corrupt it. Today the CAS `Manifest` carries an unused `VMMVersion` string and the engine threads an empty `vmmVersion`. This PR stamps the snapshot format version, the producing Firecracker version, the CPU model, and the kernel/config identity into every manifest at build time, and refuses to restore an incompatible snapshot on load with an actionable error, alongside the existing digest verify-on-load. The result: an engine either restores a snapshot it knows is compatible, or it fails closed with a message telling the operator exactly why and what to do (rebuild the template, pin the CPU, upgrade the engine).

**Honesty constraint:** this is a boring-failure-behavior and fork-correctness property (issue principles 4 and 1): an incompatible restore is a crash/corruption hazard, and we now refuse it rather than attempt it. The threat model and fork-correctness docs are updated in the same PR. The compatibility policy is exact-match for the Firecracker version and the CPU model in v1 (the conservative, correct default given Firecracker's guarantees); CPU templates that would let snapshots restore across CPU models are documented as a follow-up, not claimed. Pre-contract snapshots (built before this PR, with an empty format version) are treated as format version 0 and refused-with-a-clear-rebuild-message under the default policy, since pre-1.0 snapshots are rebuilt routinely; this is stated.

**Architecture:** Extend the CAS `Manifest` with `SnapshotFormatVersion int`, `CPUModel string`, `KernelVersion string`, and `ConfigHash string` (the existing `VMMVersion` is reused), and extend `Canonical()` to encode them deterministically (they become part of the content-addressed digest, which is correct: a snapshot's identity includes the environment that produced it). A new `internal/snapcompat` package defines the current `Environment` (format version this build supports, the detected Firecracker version, CPU model, kernel) and `Check(m Manifest, env Environment) error` returning actionable errors per mismatch. The engine captures the environment at template-build time, stamps it into the manifest, and calls `Check` on load (in the verify path) before restoring, gated by an `--allow-incompatible-snapshots` dev escape hatch mirroring `--allow-unverified-snapshots`.

**Context for the implementer:**
- `internal/cas/manifest.go`: `Manifest{Files, VMMVersion, CreatedUnix}` and the hand-rolled deterministic `Canonical()` (fixed field order, sorted files) which feeds the manifest digest. Any new field MUST be added to `Canonical()` in a fixed position with the existing `writeJSONInt`/`writeJSONString` helpers, and to any JSON (de)serialization, keeping determinism.
- `internal/fork/engine.go`: the engine has `vmmVersion` (threaded into manifests, "empty for now"), `allowUnverified` + `unverifiedWarned` (the verify-on-load escape-hatch + once-per-template warning pattern to mirror), `ensureVerified`/`isVerified` (the verified marker), `templateDigests`. The snapshot is built at template create (the manifest is written there) and verified/loaded at Fork. EngineOpts carries AllowUnverified; add AllowIncompatible + the Environment.
- Firecracker version: query `firecracker --version` (the engine knows the firecracker binary path); CPU model: read `/proc/cpuinfo` model name (or the FC machine config / CPU template id if set); kernel version: `uname -r` or the kernel image identity. Keep capture in one place so the build and the load-time current-environment agree on how they are derived.
- KVM CI verify-on-load precedent: `.github/workflows/kvm-test.yaml` already has a CAS-integrity / verify-on-load phase that tampers a snapshot and asserts the load is refused; mirror it for the compatibility check (`cmd/cas-verify` or a small helper).

### Task 1: `internal/snapcompat` contract + manifest fields

**Files:** `internal/cas/manifest.go` (+ test), create `internal/snapcompat/compat.go` + `_test.go`.

- [ ] Manifest: add `SnapshotFormatVersion int`, `CPUModel string`, `KernelVersion string`, `ConfigHash string`. Extend `Canonical()` to encode all four in a fixed position (deterministic), and update any JSON marshal/unmarshal (`WriteJSON`/read) and the manifest round-trip test. Add a `CurrentSnapshotFormatVersion` const (start at 1).
- [ ] `snapcompat/compat.go`: `type Environment struct { FormatVersions []int; VMMVersion, CPUModel, KernelVersion string }` (FormatVersions = the set this build can restore, e.g. {1}). `func Check(m cas.Manifest, env Environment) error` returning a typed, actionable error for each mismatch:
  - format version: if `m.SnapshotFormatVersion` not in `env.FormatVersions` -> "snapshot format version %d is not supported by this build (supports %v); rebuild the template" (a zero/empty version is format 0 = pre-contract -> the same refusal with a "rebuild the template; it predates the snapshot compatibility contract" message).
  - VMM: if `m.VMMVersion != env.VMMVersion` -> "snapshot was produced by Firecracker %q but this node runs %q; Firecracker does not guarantee cross-version snapshot restore, so rebuild the template on this node or pin the node to the producing version".
  - CPU model: if `m.CPUModel != env.CPUModel` -> "snapshot was captured on CPU %q but this node is %q; cross-CPU-model restore is unsafe without a CPU template, so schedule the fork on a matching CPU or rebuild".
  - kernel: a mismatch is a WARNING-level note in the error detail but not by itself fatal in v1 IF VMM+CPU+format match (document the choice: the kernel is part of the snapshot image so a kernel mismatch usually implies a different snapshot; treat a differing recorded kernel as informational unless you judge it should be fatal, then make it fatal and document). Pick one and document.
  - Return nil when compatible. Use a sentinel/typed error (e.g. `ErrIncompatible`) wrapping the detail so callers can distinguish.
- [ ] Tests: each mismatch returns a non-nil error whose message names both sides and the remediation; a fully matching environment returns nil; a zero format version is refused with the pre-contract message; `errors.Is(err, snapcompat.ErrIncompatible)` holds. Manifest Canonical determinism with the new fields (same content -> same bytes; the new fields change the digest vs an old manifest, which is expected).
- [ ] Commit `feat: snapshot format version and compatibility contract (snapcompat)`.

### Task 2: capture at build, enforce on load

**Files:** `internal/fork/engine.go`, `internal/firecracker/*` (version/CPU capture helper), `cmd/forkd/main.go`, tests.

- [ ] A capture helper (in internal/firecracker or internal/snapcompat): `DetectEnvironment(firecrackerPath string) (snapcompat.Environment, error)` running `firecracker --version` (parse the version), reading the CPU model (`/proc/cpuinfo`), and the kernel version; and a `ConfigHash` over the microvm machine config (vcpus, mem, the kernel/rootfs identity) computed at build. Inject the command runner / file reads so it is unit-testable on darwin (a fake returning canned version/cpu strings).
- [ ] Template build (engine CreateTemplate / where the manifest is written): populate `SnapshotFormatVersion = CurrentSnapshotFormatVersion`, `VMMVersion`, `CPUModel`, `KernelVersion`, `ConfigHash` from the detected environment. Replace the empty `vmmVersion` threading with the real detected value.
- [ ] Load enforcement (engine Fork / the verify path, after the digest verify): load the manifest, call `snapcompat.Check(manifest, currentEnv)`. On `ErrIncompatible`, REFUSE the fork with the actionable error UNLESS `AllowIncompatible` is set (dev escape hatch), in which case emit a loud once-per-template warning (mirror `allowUnverified`/`unverifiedWarned`) and proceed. The current environment is detected once at engine start and held on the engine.
- [ ] EngineOpts: `AllowIncompatible bool` and the detected `Environment` (or detect it in NewEngine). forkd flag `--allow-incompatible-snapshots` (default false) wired through, documented as development-only like `--allow-unverified-snapshots`.
- [ ] TDD (no KVM): with a fake environment, a manifest stamped with a mismatched VMMVersion (or CPUModel, or format version) makes Fork refuse with the actionable error and NOT launch Firecracker (seam the verify/check before the FC launch, as the existing verify test does); a matching manifest proceeds; AllowIncompatible turns the refusal into a warning-and-proceed. DetectEnvironment parses a canned `firecracker --version` + cpuinfo correctly. Keep existing engine/cas tests green (note: existing tests that build a manifest now need the format version stamped; update them).
