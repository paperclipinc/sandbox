# Snapshot Distribution (build once, pull to peers) Implementation Plan (issue #14)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Advance issue #14. Today the pool reconciler builds a template on EVERY node that lacks it by booting a VM and snapshotting (`createSnapshotsOnNodes` calls `CreateTemplate` per node), so a pool of N nodes pays N full builds and a scale-out node is slow to warm. This PR builds the template ONCE on a builder node and DISTRIBUTES it to the others by pulling the content-addressed chunks: a peer node materializes the snapshot from a holder node's CAS over the network (the CAS store, chunking, `HTTPTransport`, and `Pull`/`Materialize` already exist from PR #9), then verifies it (the existing digest verify + snapshot-compat check) before serving forks. This pairs with the capacity-aware scheduler (#17): a cold scale-out node can be seeded by a pull instead of a rebuild. Verified in CI by running TWO forkd processes on one machine: node A holds a template and serves its CAS; node B pulls the template from A, materializes and verifies it, and (on KVM) forks and execs from the pulled snapshot. No two-node cluster is needed; the CAS transport is HTTP and the pull is a real cross-process transfer.

**Honesty constraint and scope:** distribution applies to NON-encrypted templates. An encrypted template (#31) lives inside a per-scope LUKS container, so its CAS chunks would be plaintext, which would defeat encryption at rest; encrypted templates therefore stay on the per-node build path, and this is stated (the "encrypt the CAS chunk store" follow-up tracked under #31 is the prerequisite for distributing encrypted templates). The CAS-serving endpoint is access-controlled: a peer pull presents a controller-minted, template-scoped bearer token over forkd's TLS, so an unauthorized party cannot enumerate or pull template content. Every pulled chunk is verified against its digest and the whole snapshot against the recorded manifest digest + the snapshot-compat contract (#32) before the marker is written, so a malicious or corrupt source cannot inject a bad snapshot (it fails the verify, fails closed). We do not claim a measured cross-node transfer rate (no multi-node testbed); the CI proves correctness of the pull + materialize + verify + fork on one machine.

**Architecture:** forkd serves its CAS over the existing TLS HTTP server using `cas.NewHTTPHandler` (GET /cas/manifest/{digest}, POST /cas/has, GET /cas/chunk/{digest}), gated by a controller-minted bearer token validated per request. A new `PullTemplate` gRPC RPC takes {template_id, manifest_digest, source_url, pull_token} and uses `cas.Pull` (via `HTTPTransport` to source_url with the token) to fetch the manifest + missing chunks into the local CAS, then `Materialize`s the snapshot files (mem, vmstate, rootfs, volumes) into `<dataDir>/templates/<id>/`, runs the existing verify (digest) + snapshot-compat check, and writes the verified marker. The controller's pool reconciler builds the template on the first eligible node (CreateTemplate), then for the remaining nodes that lack it issues `PullTemplate` sourced from a node that already has it (found via the registry's `TemplateDigests`), minting a scoped pull token and passing the holder's CAS URL. Encrypted templates skip distribution (per-node CreateTemplate as today).

**Context for the implementer:**
- CAS: `internal/cas/http_transport.go` has `NewHTTPHandler(store)` (the server: /cas/manifest, /cas/has, /cas/chunk, digest-allowlist-validated) and `HTTPTransport` + `NewHTTPTransport(baseURL, client)` (the client: HasChunks/GetChunk/GetManifest). `internal/cas/transport.go` has `Pull` (incremental, verifies on receipt) and the store `Materialize` (reconstructs a file from chunks, removes a partial on mismatch). `PutSnapshot` chunks a snapshot into the CAS at build (already called by CreateTemplate per PR #9). `Digest.Validate()` is the path-injection barrier.
- forkd: `internal/daemon/server.go` runs the HTTP sandbox API (:9091) and the gRPC service (:9090, mTLS). `cmd/forkd/main.go` flags incl `--cas-dir`, the TLS flags, the mTLS server. The engine (`internal/fork/engine.go`) has the CAS store, the verify path (`ensureVerified`/the snapshot-compat `ensureCompatible`), the template dir layout, and reports `TemplateDigests` in GetCapacity.
- Controller: `internal/controller/sandboxpool_controller.go` `createSnapshotsOnNodes` (the per-node CreateTemplate loop; note the encryption-key fail-closed guard already there), `NodeRegistry` (`NodesWithTemplate`, `TemplateDigests`, the per-node CAS/HTTP endpoint - check NodeInfo for an address to build the CAS URL; the discovery knows the pod IP). proto `forkd.proto`.

### Task 1: forkd serves CAS (token-gated) and a PullTemplate RPC

**Files:** `proto/forkd.proto` (+ regen), `internal/daemon/server.go` (mount the CAS handler, token check), `internal/daemon/grpc_service.go` (PullTemplate handler), `internal/fork/engine.go` (a PullTemplate engine method: pull -> materialize -> verify -> marker), `cmd/forkd/main.go`, tests.

- [ ] CAS serving: mount `cas.NewHTTPHandler(engine.CASStore())` under `/cas/` on forkd's TLS HTTP server, wrapped in a middleware that requires a valid pull token (a bearer token; validate against a token the forkd was given or one the controller minted and the request carries; simplest defensible model: forkd accepts a token signed/symmetric-shared via the controller, OR a per-node static peer token the controller distributes and includes in PullTemplate; pick the simplest that is not "open to the world" and document). Serve only over TLS (refuse if forkd has no TLS, like the encryption guard). The chunks are digest-addressed and the puller verifies them, so integrity does not depend on the channel, but the token gates enumeration/pull.
- [ ] Proto: `rpc PullTemplate(PullTemplateRequest) returns (PullTemplateResponse);` with `PullTemplateRequest{ string template_id; string manifest_digest; string source_url; string pull_token; }`. Regen, commit generated code. pull_token is a credential: comment that it must not be logged.
- [ ] Engine `PullTemplate(ctx, templateID, manifestDigest, sourceURL, token string) error`: build an `HTTPTransport` to sourceURL (with the token on requests), `cas.Pull` the manifest + missing chunks into the local CAS, `Materialize` the snapshot files (mem, vmstate, rootfs, and any volume seeds) into `<dataDir>/templates/<templateID>/` per the manifest's file entries, then run the SAME verify (digest) + snapshot-compat check the fork path uses and write the verified marker. On any verify failure, remove the partial template dir and return an error (fail closed: a bad pull never becomes a servable template). Record the template digest so GetCapacity reports it.
- [ ] Daemon PullTemplate handler: call engine.PullTemplate; do NOT log the token; map errors to gRPC status.
- [ ] cmd/forkd: a `--peer-token` (or the mechanism chosen) for CAS-serving auth; mount the CAS handler when TLS is configured; document.
- [ ] TDD: TWO in-process stores/servers - serve store A via NewHTTPHandler+token middleware on an httptest TLS (or plain for the unit, with a separate token test), and engine.PullTemplate (or a thin puller) against it: a valid token + correct digest pulls + materializes + verifies; a wrong token is rejected (401/403); a tampered chunk/manifest fails the verify and leaves no marker (fail closed); pulling a digest that does not match the manifest fails. The engine PullTemplate verify-fail path is unit-testable with a fake source serving bad bytes. No KVM for the unit layer.
- [ ] Commit `feat: forkd serves its CAS and pulls templates from a peer`.

### Task 2: controller builds once and distributes

**Files:** `internal/controller/sandboxpool_controller.go`, `internal/controller/node_registry.go` (the holder CAS URL + token mint), `internal/controller/forkd_client.go` (a PullTemplate client call), tests.

- [ ] Registry: expose, per node, the CAS-serving base URL (derive from the node's address/pod IP + the forkd HTTP port + /cas) and a way to mint a pull token the holder will accept (the controller and forkd share the token mechanism from Task 1; the controller mints/passes it). `NodesWithTemplate(templateID)` already finds holders; pick a healthy holder as the source.
- [ ] Pool reconcile (`createSnapshotsOnNodes`): change the distribution policy. If the template is encrypted -> keep the per-node CreateTemplate path (document why: CAS chunks would be plaintext). Else: ensure the template is BUILT on at least one node (CreateTemplate on a first eligible node if no node has it yet); then for the remaining nodes in the deficit that lack it, issue `PullTemplate` sourced from a holder (the holder's CAS URL + a minted token + the manifest digest from the holder's TemplateDigests) instead of CreateTemplate. Track created+pulled toward the deficit. Keep the encryption-key fail-closed guard for the build path.
- [ ] forkd_client: a `pullTemplateOnNode(ctx, node, templateID, digest, sourceURL, token)` calling the PullTemplate RPC over the node's mTLS gRPC; never log the token.
- [ ] TDD (envtest + unit): with a fake forkd registry where node A holds template T (reports its digest) and node B does not, the pool reconcile issues a PullTemplate to B sourced from A (the fake records the PullTemplate call with A's URL + the digest), NOT a second CreateTemplate; an encrypted template still uses CreateTemplate on each node (no pull); when NO node has the template yet, one CreateTemplate happens (the build) then pulls for the rest. Assert the token is not logged.
- [ ] Commit `feat: pool reconciler builds a template once and distributes by pull`.
