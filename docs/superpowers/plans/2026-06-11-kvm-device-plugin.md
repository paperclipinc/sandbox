# Husk Pods: /dev/kvm Device Plugin (issue #18, PR4)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the Kubernetes device plugin that lets a pod obtain `/dev/kvm` (and `/dev/net/tun`) as a SCHEDULABLE resource request rather than via `privileged: true` + a host-path mount. This is a prerequisite for husk pods (#18): a husk pod must pass PSA `restricted` except a documented device exception, which means no privileged container and no raw host-path device mount. The plugin advertises `agentrun.dev/kvm` on nodes that actually have `/dev/kvm`, so the scheduler only places KVM-needing pods on KVM-capable nodes (scheduler truth, no bespoke node labels needed), and on Allocate it injects the device nodes into the container. Verified: the device-plugin gRPC (Registration handshake, ListAndWatch device advertisement, Allocate device injection) is unit-tested, and a kind e2e confirms the plugin DaemonSet runs and registers with the kubelet and advertises the correct count (zero on a no-KVM kind node, which is honest scheduler truth: a pod requesting the resource stays Pending where there is no KVM). The full path (a pod on a real KVM node receives /dev/kvm) needs a KVM-capable kubelet, which neither kind (no KVM) nor the Firecracker CI (no kubelet) provides; that is documented as needing the reference-node testbed.

**Honesty constraint:** the plugin advertises a device ONLY where `/dev/kvm` exists (count = a configurable number of allocatable slots, default sized to the node, or a fixed pool); on a node without `/dev/kvm` it advertises zero, so the scheduler honestly will not place KVM pods there (no double-booking, no fake capacity). The PSA story is honest: a pod using the device plugin still needs the device exception (the device plugin injects a device node, which `restricted` PSA allows via the device plugin path, NOT via `privileged`), and the doc states exactly which PSA controls are satisfied and which need the documented exception. We do NOT claim the full kubelet integration is CI-verified on a KVM node (it is not; the gRPC + registration + a no-KVM kind registration are what CI proves). No number for allocatable KVM slots is asserted as measured; it is a configurable policy.

**Architecture:** `internal/deviceplugin` implements the kubelet device-plugin v1beta1 gRPC API: a `Registrar` that registers the plugin's unix socket with the kubelet (`/var/lib/kubelet/device-plugins/kubelet.sock`), and a `Plugin` serving `ListAndWatch` (streams the current device list, re-streaming on health changes) and `Allocate` (returns a `ContainerAllocateResponse` with `Devices` = the `/dev/kvm` (+ optionally `/dev/net/tun`) `DeviceSpec`). Device discovery checks `/dev/kvm` existence and yields N allocatable slot IDs (a configurable count, since one /dev/kvm char device can back many VMs; the slot count is the admission knob). `cmd/kvm-device-plugin` is the binary (a DaemonSet on KVM nodes). `deploy/device-plugin/` is the manifest. The device-plugin API types come from `k8s.io/kubelet/pkg/apis/deviceplugin/v1beta1` (v0.32.x, matching the k8s.io/api already in use; verify it does not bump the go directive).

**Context for the implementer:**
- The device-plugin v1beta1 API: `Registration` service (the plugin calls the kubelet's Register with {Version, Endpoint (the plugin's socket name), ResourceName}); `DevicePlugin` service the plugin serves: `GetDevicePluginOptions`, `ListAndWatch` (server-streaming Device list), `Allocate` (AllocateRequest with container device IDs -> AllocateResponse with ContainerAllocateResponse{Devices: []*DeviceSpec{HostPath, ContainerPath, Permissions}}), `GetPreferredAllocation`, `PreStartContainer`. Implement ListAndWatch + Allocate fully; the others can be minimal/no-op.
- `k8s.io/kubelet/pkg/apis/deviceplugin/v1beta1` provides the gRPC types + `DevicePluginPath` (/var/lib/kubelet/device-plugins/) + `KubeletSocket`. `go get k8s.io/kubelet@v0.32.x` (match the k8s.io/api version already in go.mod; verify go 1.24 not regressed).
- Current /dev/kvm access: `deploy/daemon/daemonset.yaml` mounts /dev/kvm via a hostPath + the `agentrun.dev/kvm=true` nodeSelector. The device plugin is the PSA-restricted alternative; this PR ships the plugin (the husk pods + a forkd migration to request the resource are follow-ups; do NOT rip out the forkd hostPath in this PR, just add the plugin and document the migration path).
- ResourceName: use `agentrun.dev/kvm` (matches the existing node-label convention).
