# Husk Pods: Raw-forkd vs Pod-native Benchmark (issue #18, PR10, migration slice 5)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Honestly quantify the pod-native (husk) claim-to-first-exec cost versus raw-forkd, per issue #18's constraint "P99 claim to first-exec <= 10ms from a warm pool on the reference node; before/after in BENCHMARKS.md; if pod-native costs >2-3x raw-forkd, quantify exactly where". The data already exists in CI: the benchmark program (#15) measures raw-forkd fork-to-first-exec, and the husk-stub activation phase (#66) measures pod-native in-place activation. This slice adds a BENCHMARKS.md comparison section that synthesizes the two, states the design win (pre-scheduled dormant husk pods keep pod scheduling AND the Firecracker process spawn OFF the claim hot path, so claim-time is just snapshot activation), and is explicit that <=10ms is the bare-metal reference-node TARGET (#16, needs hardware), not a shared-CI claim. No new product code; no fabricated bare-metal numbers.

**Honesty constraint:** every number cited is the existing shared-CI-class measurement (the fork_to_first_exec P50/P99 from the bench phase; the husk-stub activation P50/P99 from the activation phase), labeled shared-CI and reproducible from those CI phases. The <=10ms bare-metal warm-pool claim-to-first-exec is a TARGET requiring the pinned reference node (#16), stated as such, never presented as achieved. The comparison is apples-to-apples about WHAT is on the claim hot path in each mode (raw-forkd: FC spawn + snapshot load + guest-ready; pod-native: snapshot activation on a pre-started dormant FC + the controller mTLS control round-trip, with pod scheduling and FC spawn amortized to warm-pool-fill time). If pod-native is slower on any component, say so and where; the honest finding is expected to be that pod-native is competitive-or-faster on the claim hot path because the FC spawn is pre-paid, while the warm-pool-fill cost (pod schedule + FC dormant start, seconds) is paid once per warm slot off the hot path.

- The design: a husk pod is pre-scheduled and runs a DORMANT Firecracker (Prepare); claim-time = Activate (snapshot load + resume + guest-ready) over the mTLS control channel. So pod scheduling (seconds) and the FC process spawn are NOT on the claim hot path (they happen at warm-pool-fill). Raw-forkd does FC spawn + load + guest-ready all at claim time (the ~69ms fork-to-first-exec).
