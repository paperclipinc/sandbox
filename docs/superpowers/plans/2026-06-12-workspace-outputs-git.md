# Workspace and State: Outputs Extraction and Git Rendezvous (W4, slice 3)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make terminate-with-outputs real, and stand up git as the merge layer for fork-and-merge. Slice 2 dehydrates the WHOLE `/workspace` into a revision on terminate; this slice adds (1) OUTPUTS EXTRACTION: a `spec.outputs` list on the claim/sandbox so terminate captures ONLY the listed paths into the revision, with an optional `{diff: true}` that records the changed paths and content hashes versus the parent revision; and (2) GIT RENDEZVOUS: an output of the form `{git: {remote, branch}}` renders the workspace's declared repo paths (`spec.git.paths`) into a git commit on a templated branch and pushes it to a shared rendezvous remote, so multiple forked attempts merge THROUGH git (git is the merge layer; the engine never does a filesystem merge). Verified: outputs capture only the listed paths (envtest with a fake transfer + unit on the filter); the diff is the changed-paths + content-hash set computed from the parent and child CAS manifests; the git rendezvous commits the repo paths and pushes to a local bare repo (integration test) on a templated branch. The whole-workspace dehydrate (slice 2) remains the default when no outputs are specified.

**Honesty constraint:** the data movement (reading the listed paths out of a booted VM) reuses the slice-2 bulk transfer and is proven on KVM there; this slice proves the OUTPUTS FILTER + the DIFF + the GIT push at the unit/integration level (a local bare repo for the rendezvous, no external git server) and the lifecycle wiring in envtest with the transfer seam. No claim that a real remote rendezvous works that a test does not show; the rendezvous remote URL is taken from the spec and the push is to whatever git URL is configured (tested against a local bare repo). Git is explicitly the ONLY merge mechanism: the engine renders repo paths into commits and pushes branches; it NEVER merges working trees itself, and the docs say a human or CI merges the branches with git. The diff is content-addressed (changed = a manifest entry whose chunk digests differ), honest about renames (a rename shows as a delete + add unless git handles it on the repo-paths side). Secrets stay excluded (the slice-2 exclude list still applies to outputs).

**Architecture:** `SandboxClaimSpec.Outputs []OutputSpec` where OutputSpec is one of: `{Path string}` (capture this path subtree), `{Diff bool}` (record a diff vs the parent revision), `{Git *GitOutput{Remote, Branch}}` (push the repo paths to a remote). On terminate with outputs set: the dehydrate captures only the union of the `Path` outputs (instead of all of `/workspace`), producing a revision whose manifest contains only those paths; if `{diff: true}`, compute the diff (changed paths + content hashes) between the parent revision's manifest and the new one and store it on the revision status / as a small diff artifact; for each `{git}` output, materialize the workspace `spec.git.paths` into a temp git worktree, commit on the rendered branch (template `{{.name}}` -> the claim/sandbox name), and `git push` to the remote URL. A new `internal/workspace/outputs.go` (filter + diff) and `internal/workspace/git.go` (the rendezvous: init/commit/push via the git CLI or go-git) hold the logic. The WorkspaceRevision gains a `diff` summary + the git push is recorded in status/an Event.
