# Husk Pods: Kubernetes Conformance Suite (issue #18, PR8, migration slice 4)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Prove the Kubernetes-conformance acceptance criteria of issue #18, each as a test, now that sandboxes are pods by default (slice 3). These criteria are OBJECT-level (admission, scheduling, quota, NetworkPolicy attachment, kubectl visibility), so they hold on the kind runner even though the nested Firecracker VMM does not boot inside a kind node (the documented kind boundary): the husk POD object is created, admitted, scheduled, accounted, and policed by Kubernetes regardless of whether its in-container VMM comes up. This slice adds a `kind-e2e-husk-conformance` set of assertions and the honest design reconciliation for networking:
1. Scheduler truth: the husk pod's sandbox cpu/memory show as ordinary pod resource REQUESTS; the scheduler accounts them against node allocatable (no double-booking; a pod that does not fit stays Pending).
2. ResourceQuota + LimitRange bound sandbox consumption with ZERO custom code (because husk pods are ordinary pods with requests): a ResourceQuota in the namespace caps the husk pods, and a pod over the quota is rejected by the quota admission, not by us.
3. NetworkPolicy governs sandbox traffic via the pod netns: a NetworkPolicy attaches to and selects the husk pod; document the reconciliation with the bespoke host-nftables egress engine (in husk mode the VM tap is in the pod netns so NetworkPolicy is the governing layer; the host-nftables egress is the raw-forkd-mode mechanism), so we do not double-enforce or imply both apply.
4. PSA restricted: the husk pod is admitted into a namespace labeled Pod Security `restricted` EXCEPT the single documented device exception (RunAsNonRoot false for /dev/kvm via the device plugin); no privileged container.
5. kubectl get pods shows sandboxes; the husk pod logs carry the stub (and, where the VMM boots, the guest console).

**Honesty constraint:** these are object-level conformance facts that kind can prove. We do NOT claim the in-VM traffic is governed by NetworkPolicy on kind (the VMM does not boot there); we prove the NetworkPolicy SELECTS and attaches to the husk pod (object-level) and document that the in-pod-netns enforcement of the VM tap is exercised on a KVM-capable kubelet (the bare-metal reference node). The PSA assertion is exact: the husk pod passes `restricted` except the named device exception, and we state which `restricted` controls it satisfies and which one it needs the exception for. The scheduler-truth and ResourceQuota facts are genuine (Kubernetes does the accounting; we add zero custom code), proven by creating the objects and asserting the native behavior. No conformance claim is made that a kind test does not demonstrate at the object level.

**Architecture:** A `kind-e2e-husk-conformance` job (or assertions appended to the slice-3 `kind-e2e-husk` job) deploys the husk-default stack and exercises each criterion against the husk pod OBJECTS (which exist on kind): a ResourceQuota/LimitRange test, a NetworkPolicy attach test, a PSA-restricted-namespace admission test, a scheduler-truth (requests-accounted / overcommit-pends) test, and a kubectl-get-pods + logs test. The networking reconciliation is documented in `docs/husk-pods.md` + `docs/networking.md` (husk mode = pod-netns NetworkPolicy governs; raw-forkd = host-nftables egress; not both). No new product code is needed for criteria 1-2-4-5 (they are native k8s behavior on the husk pod spec); criterion 3 may need the husk pod to carry the right labels for NetworkPolicy selection (already has mitos.run/pool + mitos.run/husk) and the doc reconciliation; if the husk pod spec needs an adjustment to genuinely satisfy a criterion (e.g. a missing request, a PSA field), make the minimal spec change.
