# Workspace and State: Declarative Foundation (W4, slice 1)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Lay the declarative foundation for W4 (Workspace and state): durable, versioned, forkable agent state independent of any sandbox (the PVC-to-Pod analogy). This slice ships the `Workspace` and `WorkspaceRevision` CRDs in `mitos.run/v1alpha1` matching docs/api/v2-spec.md, and the Workspace controller that manages the revision lineage, retention, and status (head, revision count, resumable, typed conditions with observedGeneration), reusing the existing content-addressed store (`internal/cas`) as the artifact layer. A WorkspaceRevision is a content-addressed point in the revision DAG with lineage (`fromClaim` or `fromWorkspaceRevision`); revisions are owner-ref'd to their Workspace for GC and pruned by the retention policy. The hydrate/dehydrate data path (a sandbox bound to a workspace hydrates the head on start, dehydrates to a new revision on terminate), the git rendezvous, the outputs extraction, and the CloudEvents revision feed are LATER slices. Verified in envtest: a Workspace reconciles to Ready with typed conditions + observedGeneration; revisions form a lineage DAG; retention prunes old revisions; the status head/count/resumable reflect the revisions; deletion GCs the revisions.

**Honesty constraint:** this is the declarative model only; no data actually moves yet (hydrate/dehydrate is slice 2), and the docs say so. "resumable" in status is computed honestly from whether the head revision carries a paired memory-snapshot ref (set by a later slice; false until then). The CRD schema is a format-affecting decision: it follows docs/api/v2-spec.md exactly (store.objectStorageRef, git.paths, retention.revisions/minAge, grants[].serviceAccount/access; status.head/revisions/resumable/conditions); deviations are recorded. Per the compliance addendum, the Workspace-not-CSI decision is recorded as an ADR (a Workspace is not a PersistentVolume; it is a versioned content-addressed artifact, not a CSI volume). No claim about durability/forkability that an envtest or a later data-path slice does not demonstrate. Single-writer-per-revision is enforced at the model level (a revision is immutable once committed; a new write makes a new revision).

**Architecture:** Two CRDs in `api/v1alpha1`: `Workspace` (spec: store, git, retention, grants; status: head, revisions, resumable, conditions, observedGeneration) and `WorkspaceRevision` (spec: workspaceRef, source lineage {fromClaim | fromWorkspaceRevision}, contentManifest the CAS manifest digest, optional memorySnapshotRef; status: committed/conditions). The Workspace controller (`internal/controller/workspace_controller.go`) reconciles a Workspace: sets typed conditions + observedGeneration, computes status.head (the latest committed revision), status.revisions (the count), status.resumable (head has a memorySnapshotRef), enforces retention (prune revisions beyond spec.retention.revisions older than minAge, respecting lineage so a pruned revision is not an ancestor still referenced), and owner-refs revisions for GC. A WorkspaceRevision is immutable once its status is Committed (single-writer doctrine). The CAS (`internal/cas`) is the artifact store (the contentManifest digest references CAS chunks); no data movement in this slice (revisions are created with a manifest ref by a later slice; this slice manages the lifecycle + a test seam to create revisions).
