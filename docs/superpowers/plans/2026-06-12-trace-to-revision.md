# Observability: One Trace ID Through to the Workspace Revision (#8)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver the observability-acceptance item that was blocked until W1 (husk pods) and W4 (workspaces) existed: a SINGLE trace id stamped from the claim reconcile through the run path to the WorkspaceRevision it produces, so an operator can take a workspace revision and find the exact orchestrator request + activation that created it (and vice versa). The claim/fork OpenTelemetry trace already exists (`controller.reconcileClaim` -> `forkOnNode` -> `forkd.Fork`, W3C propagation, no secrets in spans). This slice (1) records the dehydrate/commit of a workspace revision as a child span on that trace, and (2) stamps the W3C trace id onto the WorkspaceRevision (an `mitos.run/trace-id` annotation) so the revision is permanently linked to the trace that produced it. Verified with in-memory span tests: a claim-with-workspace reconcile that terminates produces a revision whose annotation carries the reconcile's trace id, and a `workspace.dehydrate` span is a child of the claim reconcile span. The first-exec/guest-ready spans (the in-VM telemetry bridge) remain the documented bare-metal tail.

**Honesty constraint:** spans NEVER carry secret values (only ids, counts, names, digests, timings, per the existing rule). The trace-id annotation is the W3C trace id (a public correlation id, not a secret). We prove the propagation and the stamping with in-memory span recorders in CI (the existing observability test pattern); we do NOT claim a guest-side first-exec span that needs the in-VM telemetry bridge (that stays the bare-metal tail, documented). The trace id on the revision is honest: it is the trace active when the revision was committed; if tracing is disabled (the no-op provider), the annotation is absent (not a fake id). The link is one-directional-but-complete: the revision names its trace; the trace names the revision via the dehydrate span attribute (revision name + content digest), so both directions resolve.
