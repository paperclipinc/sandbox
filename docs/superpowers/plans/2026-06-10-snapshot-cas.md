# Content-Addressed Snapshots and Integrity Implementation Plan (issue #9, foundation of #14)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close issue #9 (snapshot integrity: content addressing, verify-on-load, publish authorization) and build the verifiable foundation of the snapshot-distribution epic #14: a content-addressed chunk store with dedup, incremental transfer, and bounded-cache eviction. The multi-node P2P orchestration and the pool-update-to-all-nodes-ready propagation numbers stay OPEN in #14 because they cannot be honestly measured without a real multi-node testbed; this plan builds the transport interface they will use and says so plainly.

**Honesty constraint:** every number this PR could state (chunk reuse ratio, dedup, transfer size) is produced by a reproducible unit test. No propagation-time numbers are claimed; the README and docs say the multi-node layer is design plus interface, not measured.

**Architecture:** A new package `internal/cas` (content-addressed storage). A snapshot artifact is the set of files {mem, vmstate, rootfs}. Each file is split into fixed-size 4 MiB chunks; each chunk is stored once under its sha256 digest (dedup across all snapshots on the node). A `Manifest` lists, per file, the ordered chunk digests plus sizes, the total size, and the producing VMM version; the manifest is canonically serialized and its own sha256 is the snapshot's content address (the digest recorded in the CRD). Reconstruction reassembles the contiguous files forkd needs to mmap. Incremental transfer sends only the chunks a peer lacks. forkd verifies a template's manifest digest before loading and refuses on mismatch, with an explicit dev-mode escape hatch. A bounded local store evicts unreferenced chunks LRU; chunks backing a materialized template are pinned.

**Tech Stack:** Go stdlib crypto/sha256, os, io; net/http + httptest for the transport; controller-gen for CRD status field; KVM CI for the tamper test.

**Context for the implementer:**
- Snapshot storage today: `internal/firecracker/template.go` writes `<dataDir>/templates/<id>/snapshot/{mem,vmstate}` and `<dataDir>/templates/<id>/rootfs.ext4`. `internal/fork/engine.go` Fork loads `<dataDir>/templates/<snapshotID>/snapshot/{mem,vmstate}` by path with no verification. `internal/firecracker/template.go` `CreateTemplate`/`ListTemplates`/`HasTemplate`/`DeleteTemplate`.
- forkd flags and engine construction: `cmd/forkd/main.go`, `internal/fork/engine.go` `NewEngine`. CreateTemplate is reached via the engine and the daemon CreateTemplate RPC.
- CRD: `api/v1alpha1/types.go` `SandboxPoolStatus` (add a per-node snapshot digest map or a pool-level template digest). Regen deepcopy + CRDs with controller-gen.
- Threat model section 5 (snapshot integrity) currently all-open; CLAUDE.md authoritative; no em/en dashes; TDD; explicit-path git add; conventional commits with the `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>` trailer; lint darwin + GOOS=linux; secret values never logged (digests and ids are safe to log).

### Task 1: `internal/cas` chunking, manifest, content-addressed store

**Files:** Create `internal/cas/chunk.go`, `internal/cas/manifest.go`, `internal/cas/store.go`, and `_test.go` for each.

- [ ] Chunking (`chunk.go`): `const ChunkSize = 4 << 20`. `Digest` = sha256 hex string type. `func digestBytes(b []byte) Digest`. `func chunkFile(path string) (chunks []ChunkRef, err error)` where `ChunkRef{Digest Digest; Size int}`, streaming the file in ChunkSize reads so memory stays bounded (multi-GB mem files). TDD: a file of 10 MiB yields 3 chunks (4+4+2); identical content yields identical digests; empty file yields zero chunks; digest is stable.
- [ ] Manifest (`manifest.go`): `type FileEntry{ Name string; Size int64; Chunks []ChunkRef }`; `type Manifest{ Files []FileEntry; VMMVersion string; CreatedUnix int64 }`. `func (m Manifest) Canonical() []byte` (deterministic JSON: sorted keys, no map nondeterminism, stable field order) and `func (m Manifest) Digest() Digest` = sha256 of Canonical. `func BuildManifest(files map[string]string, vmmVersion string, createdUnix int64) (Manifest, error)` chunks each file (name to path map). TDD: same inputs yield the same manifest digest; reordering the input map does not change the digest (canonical sorts by file name); a changed byte in any file changes the digest.
- [ ] Store (`store.go`): `type Store{ root string }` at `<root>/chunks/<digest[:2]>/<digest>` and `<root>/manifests/<digest>`. `New(root string) (*Store, error)` (mkdir). `PutSnapshot(files map[string]string, vmmVersion string, createdUnix int64) (Manifest, error)` chunks and writes each missing chunk (write-to-temp-then-rename for atomicity; skip chunks already present = dedup), then writes the manifest. `HasChunk(Digest) bool`, `HasManifest(Digest) bool`, `GetManifest(Digest) (Manifest, error)`. `Materialize(manifestDigest Digest, dstDir string) error` reconstructs each file by concatenating its chunks into `<dstDir>/<name>` (streaming, verifying each chunk digest as it reads, error on mismatch or missing chunk). `MissingChunks(m Manifest) []Digest` returns chunks the store lacks (for incremental pull). TDD: PutSnapshot two snapshots sharing most content, assert the second adds few new chunks (dedup); Materialize yields byte-identical files (compare sha256 of reconstructed vs original); Materialize fails if a chunk is corrupted on disk (tamper a chunk file, expect a verify error naming the chunk).
- [ ] Commit `feat: internal/cas content-addressed snapshot store with dedup`.

### Task 2: bounded cache with LRU eviction and pinning

**Files:** `internal/cas/store.go` (extend) or `internal/cas/evict.go`, test.

- [ ] Track chunk access time (use file mtime on access in Materialize, or a sidecar index; mtime is simplest and crash-safe). `Pin(manifestDigest Digest)`/`Unpin` maintains a set of pinned manifests (persisted as marker files under `<root>/pins/<digest>`); a chunk is pinned if any pinned manifest references it. `EvictToFit(maxBytes int64) (freed int64, err error)`: compute total chunk bytes; while over budget, delete the least-recently-used UNPINNED chunk; never delete a pinned chunk; return freed bytes. TDD: fill past budget, pin one manifest, evict, assert pinned chunks survive and unpinned LRU ones are gone and the store is under budget.
- [ ] Commit `feat: bounded CAS cache with LRU eviction and manifest pinning`.

### Task 3: verify-on-load wired into forkd, digest in CRD status (closes #9)

**Files:** `internal/firecracker/template.go`, `internal/fork/engine.go`, `cmd/forkd/main.go`, `api/v1alpha1/types.go` (+ regen), `internal/controller/sandboxpool_controller.go`, proto if a digest field is added to template/capacity reporting, threat model, tests.

- [ ] On CreateTemplate: after the snapshot files are written, call the CAS store `PutSnapshot({mem,vmstate,rootfs})` and `Pin` the manifest; record the manifest digest. The engine keeps a map templateID to manifest digest (and exposes it via GetCapacity/ListTemplates so the controller learns it). Keep the existing on-disk template layout for forkd to load (Materialize is for transfer/restore; locally the original files remain), but ALSO compute and store the digest. Simplest: store the digest alongside the template (`<dataDir>/templates/<id>/manifest.digest`).
- [ ] On Fork: before LoadSnapshot, recompute the digest of the on-disk mem+vmstate (or verify against the stored manifest via the CAS store) and compare to the recorded digest; on mismatch refuse with a clear error UNLESS `--allow-unverified-snapshots` (new forkd flag, default false) is set, in which case log a loud warning. This is the verify-on-load gate. Recomputing a multi-GB mem digest on every fork is too slow for the hot path, so: verify at template LOAD/registration time (when forkd first discovers or builds the template), not per fork; cache the verified status. Implement a `VerifyTemplate(id)` called when a template is created or first seen, storing a verified marker; Fork checks the marker exists (cheap) and refuses if absent and the flag is off. Document this design choice (verify-once-at-registration, not per-fork) in a comment and the threat model.
- [ ] CRD: add `TemplateDigest string` (or a per-node digest map) to `SandboxPoolStatus`; the pool reconciler records the digest reported by forkd so it is visible in `kubectl get sandboxpool -o yaml`. Regen.
- [ ] Publish authorization note: who may publish a snapshot to a pool. For this PR, document in the threat model that snapshots are produced only by forkd's own CreateTemplate (controller-authenticated via mTLS from PR #41); externally-supplied snapshots are not yet accepted, so the publish surface is the mTLS-gated CreateTemplate. Mark external snapshot import as future.
- [ ] Threat model section 5: content addressing and verify-on-load flip to mitigated (digest recorded in CRD status, verified at registration, refuse-on-mismatch with a flagged dev escape); publish authorization documented as mTLS-gated CreateTemplate. Keep honest residual: verification is at registration not per-fork; external snapshot import not yet supported.
- [ ] TDD: a daemon/engine test that CreateTemplate records a digest and a tampered mem file fails VerifyTemplate; a Fork refuses when the verified marker is absent and the flag is off, and proceeds when the flag is on. (These use real files in a temp dir with the mock or a thin engine seam; no KVM needed for the digest logic, only file IO.)
- [ ] Commit `feat: verify-on-load snapshot integrity with digest in pool status (#9)`.

### Task 4: transfer interface and HTTP transport (incremental)

**Files:** `internal/cas/transport.go`, `internal/cas/http_transport.go`, tests.

- [ ] `type Transport interface { HasChunks(ctx, []Digest) (present map[Digest]bool, err error); GetChunk(ctx, Digest) (io.ReadCloser, error); GetManifest(ctx, Digest) (Manifest, error) }`. `func Pull(ctx, local *Store, remote Transport, manifestDigest Digest) error`: fetch the manifest, compute MissingChunks against local, fetch only those from remote, verify each chunk digest on receipt before writing, then the manifest is materializable locally. TDD: two stores A (full) and B (empty), serve A via an in-process Transport, Pull a manifest into B, assert B fetched ONLY the missing chunks (count them via a recording transport) and can Materialize byte-identically; a second Pull of an overlapping snapshot fetches only the delta.
- [ ] HTTP transport (`http_transport.go`): an http.Handler exposing GET /cas/manifest/{digest}, POST /cas/has (body: digest list, returns present set), GET /cas/chunk/{digest}; a client implementing Transport against it. TDD with httptest: round-trip a Pull over HTTP, assert byte-identical Materialize and delta-only transfer. NOTE: this is the transport the future P2P layer will use between nodes; the node-to-node orchestration (which node pulls from which, peer discovery, the 10/50/100-node propagation timing) is OUT of scope and documented as open in #14.
- [ ] Commit `feat: CAS transfer interface and HTTP transport for incremental snapshot pull`.
