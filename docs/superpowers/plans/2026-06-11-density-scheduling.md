# Density and Capacity-Aware Scheduling Implementation Plan (issue #17)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close issue #17. Make node selection capacity-aware and density-optimal. Today `SelectNode` (internal/controller/node_registry.go) ignores the memory fields it tracks and picks the node with the FEWEST sandboxes, which is both capacity-blind (a node can be packed past its RAM and OOM) and anti-density (it spreads forks of one template across nodes, defeating CoW page-cache sharing, which only happens when forks of the same template run on the SAME node). This PR adds an admission model (a node is selected only if the new fork's projected memory fits its budget under an overcommit policy) and a bin-packing scorer (prefer the warm node that already holds the snapshot and runs forks of this template, packing until the node is full, then spilling to the next). The CoW-aware metering (#33) makes the capacity signals honest: a warm node's marginal cost for another fork of a template it already runs is just the per-fork unique set, not a whole VM. A claim that fits nowhere stays Pending and is retried, failing with an LLM-legible capacity error after a bounded wait, never silently OOMing a node.

**Honesty constraint:** the admission decision uses real reported numbers (host MemTotal minus a reserve for the node budget; the CoW-aware MemoryUsed; a per-template shared-once and average-per-fork-unique estimate from the metering report). Overcommit is a configurable policy factor that is SAFE precisely because CoW sharing means provisioned >> physical and the metering proves the physical footprint; we document the factor and its basis, and we never claim a density the metering does not support. NUMA pinning and hugepage-backed guest memory are real performance levers but need specific hardware and kernel config to verify; they are documented as targets/follow-ups, not claimed as implemented or measured.

**Architecture:** forkd's `GetCapacity` reports the host total memory (from /proc/meminfo, minus a configurable reserve) and a per-template capacity estimate (shared-once resident set + average per-fork unique) derived from the existing metering report. The controller's `NodeRegistry` gains a capacity model: `available(node) = node.MemoryTotal * overcommitFactor - node.MemoryUsed`. `SelectNode` becomes a scorer: among healthy nodes, project the new fork's marginal cost (cold node placement = template shared-once + avg unique; warm node placement = avg unique) and admit only nodes where the projection fits `available`; among admitted nodes, score to PACK (prefer a warm snapshot-holder with the most existing forks of this template that still has headroom) and break ties by most-available. If no node is admitted, return a typed `ErrNoCapacity` so the claim reconciler sets Pending, increments the pending metric, and requeues, failing with an actionable error after a bounded wait.

### Task 2: capacity-aware, density-optimal SelectNode

**Files:** `internal/controller/node_registry.go` (+ test), maybe a new `internal/controller/scheduler.go`.

- [ ] A capacity/overcommit model: `available(node) = int64(float64(node.MemoryTotal) * overcommitFactor) - node.MemoryUsed` (overcommitFactor configurable on the registry/controller, default e.g. 1.0 = no overcommit until tuned, documented; a value >1 leans on CoW). A marginal-cost projection for placing a fork of templateID on a node: if the node already runs forks of templateID (warm: it has the template estimate with fork_count>0 OR holds the snapshot and has forks) -> cost = avg_fork_unique_bytes (or a small floor); else (cold) -> cost = shared_once_bytes + avg_fork_unique_bytes (use the template estimate from ANY node that has it, or a configured default if unknown).
- [ ] `SelectNode(snapshotID, preferredNode)` becomes capacity-aware + packing:
  - preferredNode honored only if healthy AND admits (fits available); else fall through.
  - Candidates = healthy nodes; for each compute projected cost and admit only if cost <= available(node).
  - Score admitted nodes to PACK: prefer a node that already holds the snapshot and runs forks of this template (warm, dense) and still fits; among warm-fitting, prefer the one with the MOST existing forks of this template (pack) while it still has headroom; if no warm node fits, pick a cold node that fits with the most available memory (spread cold starts); break ties deterministically (node name).
  - If NO node admits, return a typed `ErrNoCapacity` (wrap a sentinel so callers can errors.Is it) with an actionable message (cluster memory exhausted under the overcommit policy; scale out or raise the factor).
- [ ] Keep the no-nodes-registered and no-healthy-nodes errors distinct from ErrNoCapacity (the reconciler treats no-healthy and no-capacity as transient Pending; a genuinely empty registry is also Pending, not a hard fail).
- [ ] Tests (node_registry_test.go): admit when a fork fits; ErrNoCapacity when every node is full; a warm snapshot-holder with capacity is preferred over an empty cold node (PACKING, the opposite of the old lowest-ActiveSandboxes spread) so CoW sharing is maximized; among two warm nodes the denser one is packed until full then spill; overcommitFactor >1 admits more; preferredNode that does not fit is bypassed; deterministic tie-break. Use constructed NodeInfo with set MemoryTotal/MemoryUsed/template estimates.
- [ ] Commit `feat: capacity-aware bin-packing node selection`.

### Task 3: claim admission, Pending, and bounded capacity failure

**Files:** `internal/controller/sandboxclaim_controller.go` (+ envtest), `internal/controller/sandboxfork_controller.go` if it selects nodes too.

- [ ] In the claim reconciler where SelectNode is called: on `ErrNoCapacity` (and the existing no-node cases), set `Phase = SandboxPending`, `recordClaimPending()`, set a clear status condition/message (LLM-legible: code `NoCapacity`, the cause, the remediation), and requeue with a backoff.
