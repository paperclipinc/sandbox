# Husk Pods: Fork-Correctness Handshake on Activate (issue #18, PR3)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the correctness gap PR2 honestly flagged: the husk stub's `Activate` performs the raw snapshot restore but does NOT yet run the per-activation fork-correctness handshake, so two VMs activated from one snapshot would share CRNG state and a frozen clock, and would have no claim-time secrets. This PR wires the SAME handshake the engine's fork path uses (`NotifyForked` with fresh `crypto/rand` entropy + a per-activation generation, which makes the guest reseed the kernel CRNG, step its wall clock, and signal userspace; plus env/secret delivery) into `Activate`, fail-closed (an activation whose handshake fails is not served, mirroring the engine reaping a VM that did not reseed). This is REQUIRED before pod-native execution can become the default. Verified in KVM CI exactly like the existing fork-correctness suite: activate two VMs from one snapshot and prove DISTINCT RNG streams + correct (stepped) clocks across the activations, and that a delivered secret/env value reaches the guest while never being logged host-side.

**Honesty constraint:** entropy bytes and secret values are NEVER logged (host or guest), never in argv, never in spans/audit (the existing code already enforces this; preserve it). The fail-closed property is real: if the `NotifyForked` reseed handshake fails, `Activate` returns not-ready and the VM is not handed out, because a guest that did not reseed shares CRNG state with its siblings (the exact rule the engine's fork path enforces). The CI proves distinct RNG across activations the same way the fork-correctness phase proves it across forks (the design's load-bearing safety property); if the streams were NOT distinct the phase fails loudly. No claim that secrets are delivered unless the CI shows a delivered value reaching the guest.

**Architecture:** Extend `internal/husk` `ActivateRequest` with the claim-time config the handshake needs: `Env map[string]string`, `Secrets map[string]string`, and (optional, for parity) the `NotifyForkedNetwork` / volume mount table. `Stub.Activate`, after `LoadSnapshot` + the guest agent becoming reachable, generates fresh `crypto/rand` entropy + a per-activation generation counter and calls `vsock.Client.NotifyForkedWithConfig(generation, entropy, network, volumes)` (the same client the engine uses) and delivers env/secrets, then declares ready; on any handshake failure it returns OK=false + error and does not serve the VM (fail closed). The entropy/generation/secrets are held only in memory and never logged. This mirrors `internal/daemon/server.go` `deliverConfig`/`notifyForked`; reuse the vsock client and the entropy-generation pattern (do not duplicate the guest-side logic, which already exists in `guest/agent/notifyforked.go`).
