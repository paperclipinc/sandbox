# Husk Pods: Claim Activates a Husk Pod (issue #18, PR6, migration slice 2)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Second slice of the husk-pods migration: a claim activates a pre-scheduled warm husk pod in place, so the sandbox runs inside a Kubernetes pod. When `--enable-husk-pods` is set, the claim reconciler picks a dormant warm husk pod from the pool, sends an Activate over a network-reachable, mTLS-authenticated control channel to that pod's stub (carrying the template snapshot reference plus the claim-time secrets/env, which run through the fork-correctness handshake from PR #67 inside Activate), and sets the claim endpoint to the activated pod so exec/files route to the in-pod sandbox. Raw-forkd remains the default (flag off) and the path is unchanged. Verified: the stub control transport (mTLS gRPC Activate served by the stub, reached by a client) is tested and exercised on KVM (activate over the network, not just the unix socket); the claim reconciler's husk-activation wiring is envtest-proven against a fake stub (the claim picks a dormant pod, activates it with the right snapshot + secrets, sets the endpoint, marks the pod claimed); and the real end-to-end claim->husk-pod activation runs in kind-e2e where feasible.

**Honesty constraint:** the control channel activates a VM with tenant secrets, so it MUST be mTLS-authenticated (reuse the internal/pki CA, like the controller<->forkd channel); an unauthenticated activate of a husk pod would let any in-cluster actor inject secrets or hijack a sandbox. The fork-correctness handshake (RNG reseed, clock step, secret delivery) from PR #67 runs on activate and fails closed (a pod that did not reseed is not handed to the claim). The snapshot is delivered to the husk pod read-only (the pod mounts the node's materialized template snapshot, placed there by the existing build/distribution machinery #10/#14); we document this and that a fully pod-native CAS-pull-into-the-pod is a refinement. Sandboxes become pods only on this path (flag on); the default is still raw-forkd until slice 3. envtest proves the wiring (no kubelet, fake stub); the real activation is kind-e2e (the runner is KVM-capable).

**Architecture:** The stub (`internal/husk` + `cmd/husk-stub`) gains a network control server: in addition to the unix socket, it serves the Activate control over gRPC (or HTTP) on a pod port, gated by mTLS using the internal/pki CA (the same trust root as forkd), so only the controller (the `controller.mitos` identity) can activate. The husk Pod spec (from slice 1) gains: the control port, an mTLS cert (mounted from the PKI Secret or issued), and a read-only mount of the node's template snapshot (hostPath to `<dataDir>/templates/<snapshotID>/`) plus the kernel, so the dormant stub can load it on activate. The claim reconciler, when `r.EnableHuskPods`, runs a husk-activation path: select a dormant warm husk pod for the claim's pool (Running + Ready + not already claimed, tracked by a label/annotation), call the control client `Activate(podIP:port, snapshotID, env, secrets, network)`, on success set `claim.Status.Endpoint` to the pod's sandbox endpoint + `Status.Node` to the pod's node and mark the pod claimed (label `mitos.run/claim=<name>`); on failure or no-dormant-pod, pend (reuse the existing Pending/backpressure path). The raw-forkd `forkOnNode` path is untouched for the flag-off default.
