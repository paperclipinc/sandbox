# Guest Networking and Host-Side Egress Policy Implementation Plan (issue #3 network identity, NetworkPolicy)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Give sandboxes real network egress with host-side enforced default-deny plus an IP:port allowlist, a fresh network identity (tap, MAC, IP) per fork so forks never collide or inherit the parent's connections, and prove reach-allowed / block-denied and distinct-identity-per-fork in KVM CI. This closes the network-identity row of the fork-correctness epic (#3) and makes the `NetworkPolicy` CRD real for IP-based allowlists. The DNS-name allowlist resolver ("api.anthropic.com:443") is explicitly a tracked follow-up (PR2), because name-based rules are bypassable without a resolver we control; this PR enforces IP:port and documents the name-based path as open.

**Honesty constraint:** egress enforcement is a security boundary, so it lives host-side in nftables on the tap path where the guest cannot influence it. The threat model states exactly what is enforced (IP:port default-deny allowlist) and what is open (name-based allowlists pending the controlled resolver). No claim that name-based rules are enforced until PR2.

**Architecture:** forkd allocates a per-sandbox network identity (a MAC and an IP from a per-node sandbox /16, plus a tap device name) from an allocator like the existing UID allocator. It creates a tap device, configures the host side (IP, up, routing/NAT for egress via the node's uplink), attaches the tap to Firecracker as a network interface with the allocated guest MAC, and installs a per-tap nftables ruleset: default-drop egress, allow established/related, allow only the configured IP:port destinations, allow DNS to the node resolver only. Each fork gets its own identity and tap, so forks never share an IP or a baked NIC; under the jailer the netns further isolates. The snapshot is taken with no live NIC (or a stub) so restored memory holds no live TCP state tied to an address; the guest agent's NotifyForked hook brings the interface up fresh. Teardown removes the tap and the nftables ruleset and frees the identity.

**Linux-only, KVM-CI-verified:** the real behavior (tap, nftables, routing) requires root on Linux and is proven in the KVM CI workflow. Pure helpers (command/arg construction, identity allocation, nftables rule rendering) are unit-tested on any platform like the jailer helpers; the exec-ing wrappers are behind Linux build tags.

The Firecracker network API is `PUT /network-interfaces/{iface_id}` with body {iface_id, guest_mac, host_dev_name}. For PR1, forkd creates the tap in the host netns and applies per-tap nftables; this is enforceable and testable. Note this layering in the threat model.

### Task 1: `internal/netconf` pure helpers, identity allocator, nftables rendering

**Files:** Create `internal/netconf/identity.go`, `internal/netconf/nftables.go`, `internal/netconf/commands.go`, and `_test.go` for each.

- [ ] Identity allocator (`identity.go`): `type Identity{ TapName string; GuestMAC string; HostIP net.IP; GuestIP net.IP }`. `type Allocator` over a configured /24 or /16 sandbox subnet and a tap-name prefix; `Acquire(sandboxID string) (Identity, error)` deterministically derives a unique tap name (e.g. `sb` + short hash or an index) and a unique IP pair (host side .1 gateway is fixed; guest gets the next free host-local address) and a locally-administered unicast MAC (set the locally-administered bit, clear multicast); thread-safe, typed exhaustion, `Release(sandboxID)`. TDD: distinct sandboxes get distinct MAC/IP/tap; MAC is locally-administered unicast; exhaustion errors; reuse after release; tap name is a valid interface name (<= 15 chars, the Linux IFNAMSIZ limit minus 1).
- [ ] nftables rendering (`nftables.go`): `func RenderEgressRuleset(tap string, guestIP net.IP, policy EgressPolicy, allow []HostPort, resolverIP net.IP) string` producing an `nft` ruleset string for a per-tap chain: default drop for traffic from guestIP on the tap, accept established/related, accept the allowlisted dst IP:port pairs, accept DNS (udp/tcp 53) to resolverIP only, drop the rest. `type HostPort{ IP net.IP; Port int }`. Parse allowlist entries: PR1 accepts only IP:port forms (e.g. `10.0.0.5:443`); a hostname:port form is parsed but recorded as UNENFORCEABLE-in-PR1 and the renderer omits it (it cannot pin a name to an IP without the resolver), returning the list of skipped name-based entries so forkd can log a clear warning. TDD: ruleset for two IP:port allows contains exactly those accepts plus the default drop; a name-based entry is reported skipped, not silently included; the ruleset is deterministic.
- [ ] Commands (`commands.go`): pure builders returning argv slices for `ip tuntap add <tap> mode tap`, `ip addr add <hostIP>/30 dev <tap>`, `ip link set <tap> up`, and the teardown `ip link del <tap>`, plus the `nft -f -` apply and the per-tap chain delete. No exec here (testable). TDD: arg shapes.

### Task 3: Firecracker NIC attach + engine wiring + fork-fresh identity

**Files:** `internal/firecracker/client.go` (SetNetwork), `internal/firecracker/types.go`, `internal/fork/engine.go`, `cmd/forkd/main.go`, tests.

- [ ] `Client.SetNetwork(ifaceID, guestMAC, hostDevName string) error` -> PUT /network-interfaces/{ifaceID}. Add to VMConfig an optional network identity; StartVM attaches the NIC when set (for template creation the NIC may be absent so the snapshot bakes no live NIC; document that templates are snapshotted WITHOUT a NIC and the NIC is attached on restore/fork). For Firecracker, a network device present at snapshot time is restored by name; to give each fork a fresh tap, the snapshot is taken with NO net device and the device is added... NOTE: Firecracker does not support hot-plugging a NIC after boot or adding one on snapshot-restore in stable releases. So the model for PR1: the TEMPLATE is booted WITH a net device bound to a placeholder tap, snapshotted; each fork creates its own tap with the SAME host_dev_name the snapshot expects but in the fork's own context. Because raw-forkd shares the host netns, the host_dev_name must differ per fork, which the snapshot bakes. This is the same constraint as the vsock UDS: the clean answer is per-VM netns (jailer/husk-pod) so the baked host_dev_name resolves to a distinct tap per netns. INVESTIGATE the actual Firecracker capability for your pinned version (v1.15) and pick the working approach; if snapshot-restore cannot rebind the tap in the host netns, then PR1 scopes networking to NON-forked sandboxes (claims that boot fresh rather than restore-from-snapshot) and documents that snapshot-fork networking requires per-VM netns (husk pods, #18). BE HONEST in the plan output about which case you implemented and why. Escalate to the controller (report NEEDS_CONTEXT) if the Firecracker capability is ambiguous rather than guessing.
