# Operator Ergonomics: kubectl-sandbox tree/logs/top/exec (#7)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Round out the `kubectl sandbox` operator plugin (ROADMAP #7). It already has `ls` (SandboxClaims) and `ps` (SandboxForks); this slice adds the operator verbs that close the "what is running, where did it come from, what is it doing" loop: `tree` (render the fork/lineage DAG, the distinctive verb from the v2 spec), `logs` (the husk stub + guest console for a sandbox), `top` (per-sandbox resource usage from the CoW-aware metering), and `exec` (operator exec into a sandbox over the sandbox API, token-scoped). All read through the existing CRD + sandbox-API surfaces; nothing new server-side. Also folds Box (box.ascii.dev) into the competitor positioning as part of the docs touch. Verified: each verb has a unit test against fake CRD objects / a fake sandbox API; `tree` renders a known lineage DAG correctly; the kind-e2e smoke exercises `ls`/`ps`/`tree` against the mock control plane (object-level).

**Honesty constraint:** these verbs READ existing data (CRDs, the sandbox HTTP API, the metering endpoint); no new claim about capability the server does not already expose. `top` reports the CoW-aware metering numbers (#33, the shared-once accounting) honestly labeled as the marginal unique cost, not raw memory.current; if a metering value is not available for a sandbox it shows a dash, not a fabricated number. `logs` surfaces the husk stub console + (where reachable) the guest console; on kind where the VMM does not boot it shows the stub console and says the guest console needs a running sandbox (the #18 boundary), not an error. `exec` is token-scoped (uses the per-sandbox bearer token from the claim's token Secret, the same gate the SDK uses); it does not bypass auth. The Box positioning note states only what box.ascii.dev publicly shows (disk-fork, SaaS-only, agent-native CLI, no published latency numbers), no invented comparison.

### Task 1: tree (lineage DAG) + top (CoW metering)

**Files:** `cmd/kubectl-sandbox/` (tree.go, top.go or extend main.go) + tests.

- [ ] `tree`: render the fork/lineage DAG for a pool/claim/sandbox. Walk the SandboxFork/Claim source refs (and owner refs) to build the parent->child tree, render it as an indented ASCII tree (like `kubectl tree` / git log --graph), showing each node's name, phase, and node. A `--pool <name>` scopes to a pool; no arg renders all. Unit test: a known set of claims/forks with source refs renders the expected tree (including a multi-level fork chain and siblings).
- [ ] `top`: per-sandbox resource usage from the CoW-aware metering (#33): for each running sandbox show the marginal unique (private-dirty) memory + the shared-once attribution, honestly labeled, plus cpu if available. A sandbox with no metering datum shows a dash. Pull the numbers from the metering source (the capacity heartbeat / a forkd endpoint; reuse the controller's NodeRegistry view if the plugin can read it, else the sandbox API). Unit test: a fake metering source renders the expected rows; a missing datum shows a dash, never a zero or a fabricated value.
- [ ] Commit `feat: kubectl sandbox tree and top operator verbs`.
