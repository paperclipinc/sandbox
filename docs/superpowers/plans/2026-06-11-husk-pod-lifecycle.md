# Husk Pods: Pod Object and Lifecycle Controller (issue #18, PR5, migration slice 1)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** First slice of the husk-pods controller migration: when `--enable-husk-pods` is set, the pool reconciler maintains a warm pool of pre-scheduled "husk" Pod objects (each running the dormant-VMM stub, requesting the `agentrun.dev/kvm` device, non-privileged), owner-referenced to the pool for GC, scaled to the pool's Replicas. This slice builds and verifies the Pod-object MANAGEMENT (create to the warm-pool size, track readiness, delete on scale-down, GC via owner reference), behind a flag, with the raw-forkd snapshot-on-nodes path UNCHANGED and still the default (flag off = exactly today). It does NOT yet wire claim activation to husk pods (slice 2) or flip the default (slice 3); it makes the husk pods exist and be managed, which the rest of the migration builds on. Verified in envtest: a pool with husk-pods enabled creates the right number of husk Pod objects with the correct spec (stub container, kvm resource request, non-privileged securityContext, owner-ref, labels), scaling Replicas up/down creates/deletes them, and the raw-forkd path is untouched when the flag is off.

**Honesty constraint:** envtest has no kubelet, so this slice proves the controller's Pod-object lifecycle logic (creation, count maintenance, owner-ref GC, scale-down), NOT that the husk pods actually RUN or activate (that needs a kubelet + the device plugin, exercised in later slices via kind-e2e). The docs say exactly that. The husk pod is "PSA restricted except the documented device exception" (the `agentrun.dev/kvm` resource gives it the device without `privileged: true`); we render the most restricted securityContext that still allows the stub to run Firecracker (no privileged; drop all caps and add back ONLY what the jailer/tap genuinely needs, documented; runAsNonRoot where the jailer permits, else documented), and we do NOT claim full PSA-restricted conformance until the conformance-suite slice proves it. Sandboxes are not pods until the migration completes and the default flips (slice 3); this slice is opt-in and additive.

**Architecture:** A new `internal/controller/huskpod.go` builds the husk Pod spec (`buildHuskPod(pool, template, opts) *corev1.Pod`) and a reconcile helper `reconcileHuskPods(ctx, pool, template) (ready int, err error)` that lists the pool's husk pods (by owner-ref + a label selector), creates pods up to `pool.Spec.Replicas`, and deletes extras on scale-down. The pool reconciler, when `r.EnableHuskPods`, drives `reconcileHuskPods` instead of `createSnapshotsOnNodes` (the raw-forkd path stays for the flag-off default). The husk Pod spec: a single container running the husk-stub image with the stub args (prepare a dormant Firecracker; serve a control endpoint), `resources.requests/limits["agentrun.dev/kvm"] = 1` (the device plugin from PR #68), a non-privileged securityContext (no privileged; drop ALL caps then add only the documented minimum; seccomp RuntimeDefault), owner-referenced to the SandboxPool, labeled `agentrun.dev/pool=<name>` + `agentrun.dev/husk=true`. A `Dockerfile.husk-stub` builds the stub image (for later kind-e2e). The kvm device resource name is the `agentrun.dev/kvm` from PR #68.

**Context for the implementer:**
- `internal/controller/sandboxpool_controller.go`: the Reconcile + the deficit logic (`readySnapshotCount`, `createSnapshotsOnNodes`, the `Replicas` desired count). The husk path mirrors the deficit shape but creates Pod objects. `SandboxPoolReconciler` is the struct to add `EnableHuskPods bool` + the husk-pod image ref + the kvm resource name to.
- `internal/controller/{pki_bootstrap.go,token_secret.go,enc_key_secret.go}`: the controller-runtime client usage + `controllerutil.SetControllerReference` (owner-ref) pattern to mirror for the husk pods.
- `cmd/controller/main.go`: the flags (add `--enable-husk-pods` default false, `--husk-stub-image`); the reconciler construction (wire the new fields).
- `cmd/husk-stub/main.go` + `internal/husk`: the stub the pod runs (its flags: --firecracker, --kernel, --control-socket or a control listen addr, --device paths). For the pod, the stub needs a network-reachable or node-reachable control channel (slice 2 wires activation; for slice 1 the pod just runs the dormant stub serving its control socket; the activation transport is slice 2, do NOT build it here, but make the pod spec carry what the stub needs to Prepare a dormant FC).
- The device plugin resource: `agentrun.dev/kvm` (PR #68, deploy/device-plugin). The husk pod requests it.
- api/v1alpha1: SandboxPool/Template. If the husk pod needs the snapshot reference, it gets it from the template (the snapshot is activated at claim time in slice 2; the dormant pod may not need the snapshot until activate, so slice 1's pod just prepares a dormant FC).
