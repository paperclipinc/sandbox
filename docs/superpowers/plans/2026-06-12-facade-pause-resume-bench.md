> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Map the upstream Sandbox pause/resume lifecycle onto our fast warm-pool re-activation, and stand up the `bench/facade/` latency comparison that is #19's headline acceptance criterion ("measures order-of-magnitude better claim latency; reproducible from bench/facade/ against the upstream reference on the same cluster"). In upstream v0.4.6 there is no stateful hibernate field; pause/resume is the `spec.replicas` 0 to 1 toggle, and their reference controller deletes the pod on replicas 0 and cold-creates a fresh pod on replicas 1. Our facade maps replicas 0 to releasing the sandbox back to the warm pool of dormant husk pods, and replicas 1 to a fast in-place re-activation (the ~42ms husk activation), so resume is an order of magnitude faster than a cold pod create. This slice: the facade pause/resume mapping (object-level proven in envtest + the conformance job), and `bench/facade/` (the head-to-head harness + methodology + the honest result). Verified: a Sandbox toggled replicas 1 to 0 to 1 drives our facade to release-then-reactivate at the object level on kind; the order-of-magnitude resume-latency claim is stated as a bare-metal-reference-node target (the husk activation latency vs an upstream cold-pod-start), since the in-VM resume cannot be measured on kind (the #18 nested-VMM boundary), with the harness reproducible on a KVM-capable kubelet.

**Honesty constraint:** the headline "order-of-magnitude better claim latency" is the acceptance criterion, so it must be honest. On kind/shared-CI the husk VMM does not boot (the documented #18 boundary), so the in-VM resume latency CANNOT be measured there; a naive head-to-head on kind would falsely show our facade WORSE (their controller cold-starts a real pod that reaches Ready; ours does not boot the VMM on kind). Therefore: the bench/facade harness deploys both controllers and measures resume latency, and is RUN on a KVM-capable kubelet (the bare-metal reference node, #16) for the real numbers; on shared-CI it measures only the object-level reconcile latency (facade vs upstream controller reconcile speed) which is NOT the headline claim. The order-of-magnitude claim is sourced from the component latencies (our husk activation ~42ms P50 shared-CI from #66 vs an upstream cold-pod-start measured by the harness or cited) and marked a bare-metal target until the reference node runs the full head-to-head. No fabricated head-to-head numbers; the harness + methodology are committed and reproducible, the bare-metal result is a target with the issue ref. Conformance is preserved: replicas 0 still shows 0 actual replicas (their tests assert that); the warm-pool release is our implementation detail behind the conformant observable.

**Architecture:** The facade SandboxReconciler (slice 1 handles replicas 1 = activate, replicas 0 = terminate). This slice refines replicas 0 to RELEASE the sandbox to the warm pool (the husk pod returns to dormant in the pool, not destroyed) and replicas 1 (after a 0) to RE-ACTIVATE a warm dormant pod fast, while keeping the conformant observable (replicas 0 => status.replicas 0, no serving endpoint; replicas 1 => Ready). The Sandbox status mirrors the run-path state. `bench/facade/` holds a harness that deploys our facade and the upstream reference controller on one cluster, applies equivalent Sandbox manifests, toggles replicas, and records claim and resume latency for each, plus a methodology doc; on shared-CI it records object-level reconcile latency and prints the bare-metal target for the in-VM resume.
