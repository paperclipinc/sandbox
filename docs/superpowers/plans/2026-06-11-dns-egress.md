# Controlled DNS Resolver for Name-Based Egress Implementation Plan (issue #47)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close issue #47. Make name-based egress allowlists real so an agent sandbox can reach `api.anthropic.com:443` by name. Today `internal/netconf` enforces only IP:port allow rules; a host that is a DNS name is parsed but omitted from the nftables chain (see `nftables.go:50` "cannot be enforced in PR1 without a controlled resolver"). This PR adds that controlled resolver: a per-node DNS proxy that the guests are pointed at, which resolves only allowlisted names and, as it resolves each, pins the resolved (ip, port) into a per-sandbox nftables timeout set that the egress chain consults. Because the proxy returns exactly the IPs it pinned, the guest and the firewall always agree on the address, with no resolve-and-pin race. Verified in KVM CI: a sandbox allowed `name:port` resolves the name through the proxy and reaches that host on that port, and is blocked for an unlisted name, for the right host on a wrong port, and for a direct IP it never resolved.

**Honesty constraint:** the threat model is updated in the same PR. Name-based egress trusts the proxy's upstream resolver and has a bounded TTL window where a previously-resolved IP remains allowed after a name stops resolving to it; both are documented. Source-IP attribution (each sandbox has a unique /30 guest IP from the identity allocator) is the mechanism the proxy uses to map a query to a sandbox; the anti-spoof saddr pinning already in the chain backs this. We state precisely what is enforced (a guest can reach an IP on a port only if it resolved an allowlisted name to that IP, or the IP:port is statically allowed) and the residual risks (a malicious upstream resolver, a name that maps to a shared CDN IP also used by a denied name, the TTL window).

**Architecture:** A new `internal/dnsproxy` package implements a DNS server (UDP and TCP on port 53) bound to a node-wide resolver IP that every sandbox chain already allows port 53 to. It keeps a registry mapping each sandbox guest IP to that sandbox's name allowlist (name to allowed ports). For each query it attributes the source by the client IP, checks the queried name against that sandbox's allowlist, and if allowed forwards to a configured upstream, then for every A record in the answer pins (recordIP, allowedPort) into that sandbox's nftables set with a timeout of max(recordTTL, floor) and returns the answer to the guest; if the name is not allowlisted it returns REFUSED and pins nothing. `internal/netconf` gains a per-sandbox `ipv4_addr . inet_service` timeout set and an egress-chain accept rule consulting it; the proxy's pinner runs `nft add element`. forkd constructs the proxy, registers/deregisters sandboxes with their name allowlist on fork/teardown, and the guest agent writes `/etc/resolv.conf` pointing at the resolver IP.

**Dependency (Task 1, verify go-1.24 first):** `github.com/miekg/dns` is the standard Go DNS library for both server and client. Before adding, confirm its `go` directive is <= 1.24 and it pulls no fragile tree (it is a mature, widely-used, go-1.x library; verify and pin). If for any reason it is disqualifying, implement the minimal DNS message parse/serialize for A/AAAA queries directly (well-specified but more work); prefer miekg/dns.

**Context for the implementer:**
- `internal/netconf/nftables.go`: `RenderSandboxChain(tap, guestIP, policy, allow []HostPort, resolverIP net.IP)` already allows established, static IP:port accepts, and udp/tcp 53 to resolverIP, then the final verdict; `ParseHostPort`/the allow parsing already separates a DNS name (isName=true) from an IP. `RenderSharedTable`/`SandboxChainName(tap)`/`DispatchMapName`/`SharedTableName` are the table primitives. The per-tap chain is reached via an interface-vmap dispatch and ends in a terminal verdict for this sandbox only.
- `internal/network` + `internal/netconf/commands.go`: how the chain is applied/torn down (nft -f -), the Manager/runner injection for testability on darwin.
- `internal/fork/engine.go`: `prepareForkNetwork`/`teardownForkNetwork`, the identity allocator giving each sandbox a unique /30 guestIP and a resolverIP slot (engine.go:56 notes resolverIP is already an optional plumbed value).
- Guest agent: `guest/agent/main.go` configure/NotifyForked handlers; add writing `/etc/resolv.conf` with `nameserver <resolverIP>`.
- The egress allow source: SandboxTemplate egress `allow` entries (api/v1alpha1) that are names (host without an IP) carry name:port; the netconf parse distinguishes them.
- KVM CI networking precedent: `.github/workflows/kvm-test.yaml` networking phase uses a net-smoke server at 198.51.100.2:8080 (allowed) vs :9090 (denied), and needs `iptables -I DOCKER-USER -j ACCEPT` so the GitHub runner Docker FORWARD policy does not mask our inet table. `cmd/net-smoke` and `cmd/test-agent` (egress mode) are the helpers.

### Task 1: `internal/dnsproxy` resolver with allowlist enforcement and pinning

**Files:** Create `internal/dnsproxy/proxy.go`, `internal/dnsproxy/registry.go`, `internal/dnsproxy/pinner.go`, and `_test.go`.

- [ ] Verify and add `github.com/miekg/dns` (go-1.24-clean); else implement the minimal A-record DNS subset and document.
- [ ] `registry.go`: `type Registry` mapping a sandbox guest IP (net.IP, the source of queries) to `Allowlist` (map of lowercased FQDN to a set of allowed ports). `Register(guestIP net.IP, names map[string][]int)`, `Deregister(guestIP)`, `Lookup(guestIP, name) (ports []int, ok bool)`. Concurrency-safe. Names are matched exactly (a follow-up can add suffix wildcards; v1 is exact FQDN match, document).
- [ ] `pinner.go`: `type Pinner interface { Pin(guestIP net.IP, tap string, ip net.IP, port int, ttl time.Duration) error }`; an `NftPinner` running `nft add element inet <table> <set(tap)> { <ip> . <port> timeout <ttl>s }` via an injected runner (so unit-testable); a `FakePinner` recording pins.
- [ ] `proxy.go`: `Server{ registry, pinner, upstream string, ttlFloor time.Duration, tapFor func(guestIP) string }`. A miekg/dns Handler: parse the question (A/AAAA); identify the client IP from the ResponseWriter RemoteAddr; `registry.Lookup(clientIP, qname)`; if not ok -> reply REFUSED (and log at debug); if ok -> forward the query to upstream (a dns.Client), and for each A answer record, for each allowed port, `pinner.Pin(clientIP, tap, recordIP, port, max(record.TTL, ttlFloor))`, then write the upstream answer back to the guest. Handle upstream failure (SERVFAIL to the guest, pin nothing). `ListenAndServe(addr)` on udp and tcp. A `Shutdown`.
- [ ] Tests (with a fake upstream dns server via miekg/dns test server or a stub dns.Client, and a FakePinner): an allowlisted name resolves, returns the upstream A records, and pins each (ip, port) for each allowed port with ttl >= floor; a non-allowlisted name returns REFUSED and pins nothing; a name allowed on port 443 only pins port 443 (not other ports); the source-IP attribution maps to the right sandbox (two registered guests with different allowlists get different verdicts for the same name); upstream failure -> SERVFAIL, no pin.
- [ ] Commit `feat: internal/dnsproxy resolves allowlisted names and pins resolved IPs`.
