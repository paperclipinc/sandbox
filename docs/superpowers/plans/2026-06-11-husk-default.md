# Husk Pods: End-to-End in a Cluster and Pod-Native by Default (issue #18, PR7, migration slice 3)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the husk-pods path work end to end in a real cluster and become the default. Two things: (1) close the architectural gap that in husk mode the pool creates husk pods but nothing builds the template snapshot they mount; forkd stays as the snapshot BUILDER, so the pool in husk mode ensures the template snapshot is built on nodes (the existing build/distribution machinery) AND maintains husk pods affined to those snapshot-holding nodes, so the husk pods actually have a snapshot to activate. (2) Flip the default to pod-native, with raw-forkd (the fork-per-claim run path) behind a flag for fallback. forkd the daemon stays deployed (it builds template snapshots and serves the raw-forkd fallback). Verified in kind-e2e (the runner is KVM-capable): with pod-native default, the pool builds a template snapshot on the node, husk pods come up and mount it, a claim activates a husk pod and reaches Ready, and exec works over the pod's token-gated sandbox API. The standalone husk activation is already KVM-CI-proven (slice 2); this slice proves the full CLUSTER path with a pool-built snapshot and makes it the default.

**Honesty constraint:** the build-vs-run split is stated plainly: forkd builds template snapshots (a privileged operation on the node) and the husk pods run the sandboxes (unprivileged, device-plugin-scoped). "Raw-forkd behind a flag" means the fork-per-claim RUN path is opt-in; forkd-the-builder is not removed. The default flips ONLY because the full cluster path is proven in kind-e2e in this same PR; if the kind-e2e cannot prove the full claim->husk-pod->exec path reliably, we do NOT flip the default and say so (the flip is gated on the proof). Honest Kubernetes semantics: with the flip, sandboxes ARE pods (the husk path); we update the docs and threat model to reflect the default, and keep raw-forkd documented as the fallback.

**Architecture:** The pool reconciler in husk mode does BOTH: ensure the template snapshot is built on the husk pods' target nodes (reuse `createSnapshotsOnNodes` / the forkd CreateTemplate build path, which materializes `<dataDir>/templates/<id>/snapshot` and reports it via the node registry), THEN maintain the warm pool of husk pods with nodeAffinity/placement to nodes that HAVE the snapshot (so the read-only snapshot hostPath mount resolves). The default flag flips: `--enable-husk-pods` defaults TRUE (pod-native), and `--enable-raw-forkd` (or inverting to `--raw-forkd`) selects the fork-per-claim fallback; the claim + pool reconcilers default to the husk path. The deploy base keeps forkd (builder + fallback) + the device plugin + the husk-mode controller and ships the husk-stub image. kind-e2e drives the full default path on the KVM-capable runner.
