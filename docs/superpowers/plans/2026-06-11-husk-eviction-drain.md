# Husk Pods: Eviction, PDB, and Drain (issue #18, PR9, migration slice 4b)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Complete the husk-pods lifecycle conformance: define and test eviction, preemption, PodDisruptionBudget, and drain behavior, so a husk pod disappearing (node drain, preemption, eviction, manual delete) is handled cleanly. (1) The controller creates a PodDisruptionBudget per pool so a node drain cannot evict the whole warm pool at once. (2) The pool reconciler watches its husk pods (Owns) so a deleted husk pod is replaced (the warm pool self-heals). (3) When a CLAIMED husk pod disappears, the claim is re-pended (its sandbox is gone) and re-activates on a replacement dormant pod. (4) A pool `drainPolicy` (Kill default, Checkpoint optional) governs what happens to an active sandbox on drain: Kill terminates and re-pends; Checkpoint snapshots the live VM first (via the existing ForkRunning/CreateSnapshot primitives) so it could be restored elsewhere. (5) Pending claims produce unschedulable husk pods that cluster-autoscaler/Karpenter act on natively (the husk pods request real resources; if they do not fit they are Pending/unschedulable, a native signal), tied to the existing pending-claims metric (#17). Verified in kind-e2e at the object level (PDB exists, deleting a husk pod self-heals the pool, deleting a claimed husk pod re-pends the claim), with the live-VM Checkpoint path plumbed and documented as exercised where the VMM runs (bare-metal kubelet).

**Honesty constraint:** the object-level lifecycle (PDB, pod-delete self-heal, claim re-pend on pod loss, the drainPolicy plumbing, unschedulable-pending) is kind-testable and proven there. The Checkpoint-on-drain path (snapshot the live VM so it survives the drain) needs a running VMM, which the kind nodes cannot boot (the documented nested-VMM boundary); we implement and unit-test the checkpoint logic and document that the live snapshot is exercised on a KVM-capable kubelet. Kill+re-pend is the fully-proven baseline. We do NOT claim a sandbox survives a drain via checkpoint on kind; we prove the policy is honored at the object level and the Kill+re-pend path works, and document the Checkpoint path's runtime requirement.

**Architecture:** The pool reconciler creates a `policy/v1` PodDisruptionBudget owned by the pool (selector `mitos.run/pool=<name>,mitos.run/husk=true`, `minAvailable` derived from Replicas and a documented disruption budget). It gains `Owns(&corev1.Pod{})` so husk pod deletion triggers a pool reconcile (the deficit logic recreates the replacement). A new SandboxPool `drainPolicy` field (Kill default, Checkpoint optional). The claim reconciler re-pends a claim whose husk pod is gone: it watches husk pods (or reconciles claims on pod events) and, when the pod labeled `mitos.run/claim=<claim>` no longer exists (or is terminating), sets the claim back to Pending and clears the endpoint so the next reconcile re-activates it on a replacement dormant pod. The Checkpoint drain path uses the engine's `ForkRunning`/`CreateSnapshot` to snapshot a live sandbox before its pod is killed (plumbed; the live snapshot runs where the VMM runs).
