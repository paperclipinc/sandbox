# Husk Pods: Dormant-VMM Stub with In-Place Activation (issue #18, PR2)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the architectural heart of husk pods (#18): the dormant-VMM stub with in-place activation. Today the fork engine starts a Firecracker process AND loads the snapshot back to back at claim time. The husk-pods model pre-starts a dormant VMM inside a pre-scheduled pod, so "claim = activate" is just the snapshot-load + resume in the already-running process (in the pod's own cgroup/netns), which is what gets activation toward the <=10ms target. This PR builds a standalone stub (`cmd/husk-stub` + `internal/husk`) that: at PREPARE time starts Firecracker dormant (process + API socket up, no boot, awaiting a snapshot-load) and idles on a control channel; on an ACTIVATE control message performs `LoadSnapshot` + `Resume` + vsock wiring in-process and reports ready. Verified in KVM CI: prepare a dormant stub, send activate with the bench snapshot, exec via the guest agent, and measure the honest activation latency (load-start to first successful exec, NOT the raw restore syscall). This does NOT yet migrate controllers or run inside a real pod; it proves the prepare/activate split works and measures its latency. The full pod integration (device plugin, controller migration, conformance) follows.

**Honesty constraint:** the measured activation latency reported in BENCHMARKS.md / docs/husk-pods.md is the end-to-end load-start to first-successful-exec on the shared-CI KVM runner, consistent with the existing benchmark program's honesty rule (never the ~3ms restore syscall alone). If activation does not meet <=10ms on shared CI, we report the real number and note <=10ms is the bare-metal reference-node target (#18), not a shared-CI claim. The stub is a NEW standalone component reusing the existing firecracker client primitives; it does NOT refactor the engine's fork path (blast radius contained), and it does not claim sandboxes are pods. The control channel is local (unix socket) and carries no secrets in this PR (the snapshot path + overrides only); secret/RNG delivery on activate is the existing engine's job and is noted as the integration follow-up.

**Architecture:** `internal/husk` defines a `Stub` that wraps the firecracker client: `Prepare(ctx, cfg)` calls `firecracker.StartVM(cfg)` to bring up a dormant Firecracker (process + API socket, no InstanceStart), then serves a control protocol on a unix socket; `Activate(req)` (driven by a control message) calls `LoadSnapshotWithOverrides(mem, vmstate, resume=true, overrides)` then waits for the guest agent over vsock to be reachable, and reports `{ready, activateLatency}`. The control protocol is a tiny line-delimited JSON request/response (Activate{snapshotDir, networkOverrides?} -> ActivateResult{ok, vsockPath, latencyMs, error}). `cmd/husk-stub` is the binary (flags: --firecracker, --kernel, --workdir, --control-socket); it Prepares dormant on start and serves Activate. The stub measures activation latency = from receiving Activate to the first successful guest-agent ping/exec.

**Context for the implementer:**

The engine's Fork (`internal/fork/engine.go`) shows the full sequence (StartVM -> LoadSnapshot -> vsock -> NotifyForked -> exec-ready) to mirror the activate portion; reuse, do not refactor engine.Fork.

- `cmd/bench` shows driving the firecracker client + the vsock exec + the nearest-rank latency stats (`internal/benchstat`); reuse benchstat for the activation-latency summary.
