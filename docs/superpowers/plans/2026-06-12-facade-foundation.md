# agents.x-k8s.io Conformance Facade: Foundation (issue #19, PR12, W2 slice 1)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Lay the foundation for the W2 conformance facade (issue #19): present sandboxes via the upstream SIG agent-sandbox API (`agents.x-k8s.io`, the real group; the API is v1alpha1, not the v1beta1 the issue text guessed) on our engine, so an upstream `Sandbox` object is fulfilled by a husk pod (now that #18 made pod semantics real). This slice ships: the agent-sandbox API made available (vendored if the toolchain allows, else their CRD re-declared from the vendored YAML), a facade controller (`cmd/facade` + `internal/facade`) that maps an upstream `agents.x-k8s.io/v1alpha1 Sandbox` to our husk-backed run path and mirrors its lifecycle + status, the ADR recording the facade approach and the our-vs-their naming collision decision, and the conformance-approach doc. The full upstream conformance harness (their examples + e2e vendored and run in CI), pause/resume as memory snapshot, and the SandboxWarmPool/SandboxClaim extension mappings are later slices. Verified in envtest: an upstream Sandbox object drives our run path to Ready and a deletion terminates it.

**Honesty constraint:** "implement their API, do not fork or shadow it" (the issue) means we use their EXACT API surface. The faithful approach is to vendor their Go types (`sigs.k8s.io/agent-sandbox`); but every published version declares `go 1.26+`, which forces our toolchain from 1.24 to 1.26. Go 1.26 is current (the local toolchain is 1.26.3), so the bump is acceptable IF golangci-lint can analyze a go-1.26 module (verify this; we hit a go-version-vs-lint wall before with go 1.25). Decision rule, executed in Task 1: try the go 1.24 -> 1.26 bump + vendor their module; if golangci-lint cannot analyze go 1.26 (or the bump is otherwise blocked), FALL BACK to re-declaring the `agents.x-k8s.io/v1alpha1 Sandbox` CRD by hand from their vendored CRD YAML (which is still implementing their exact API, validated by their conformance tests in a later slice, not shadowing a different API). Whichever path, the upstream conformance e2e (a later slice) is the drift guard; this slice does NOT claim full conformance, only the foundation + the lifecycle mapping. The pause/resume "milliseconds-from-memory vs their disk-roundtrip hibernation" claim is documented as a design difference and measured later, not asserted here. Honest k8s semantics throughout.

**Architecture:** Either vendor `sigs.k8s.io/agent-sandbox/api/v1alpha1` (the core `Sandbox`) + `.../extensions/api/v1alpha1` (SandboxClaim/SandboxWarmPool/SandboxTemplate) after a go 1.26 bump, OR define `internal/facade/api/v1alpha1` matching their CRD (from the vendored CRD YAML) if the bump is blocked. `internal/facade` is a controller-runtime controller that watches `agents.x-k8s.io/v1alpha1 Sandbox` and reconciles it onto our run path: their `spec.podTemplate` maps onto our husk pod (#18); a Sandbox with replicas 1 creates/activates our husk-backed sandbox and mirrors readiness into their `status` conditions; replicas 0 / deletion terminates. `cmd/facade` runs this controller (a separate binary so the facade is opt-in and does not entangle our core controller). The bridge annotation `mitos.run/pool` links an upstream Sandbox to one of our pools (the warm-pool source); extras stay in our `mitos.run` group.

**Context for the implementer:**
- Upstream: `sigs.k8s.io/agent-sandbox` v0.4.6. The core `Sandbox` is in `api/v1alpha1/sandbox_types.go` (group `agents.x-k8s.io`, version v1alpha1; SandboxSpec = podTemplate (required) + volumeClaimTemplates + Lifecycle (inline) + replicas (0 or 1, default 1) + service; SandboxStatus = serviceFQDN + conditions + ...). The extensions (SandboxClaim, SandboxWarmPool, SandboxTemplate) are in `extensions/api/v1alpha1`. Their CRD YAML and examples/ and test/e2e are in the module (vendor the CRD YAML + examples for later conformance). The go directive is 1.26.2.
- Our run path (#18): `internal/controller` (SandboxPool, SandboxClaim, the husk pod activation); a Sandbox -> our claim -> husk pod. The facade creates/owns our objects (or calls the same path). `internal/controller/huskpod.go` (the husk pod from a pool), the claim activation.
- Our toolchain: go 1.24 (go.mod, ci.yaml + kvm-test.yaml setup-go, Dockerfile.* golang:1.24). golangci-lint is pinned/latest in the lint job. The go-version-vs-lint wall (a dep needing a higher go than golangci-lint was built with fails the lint) is the risk; we resolved it before by pinning to the lower go.
