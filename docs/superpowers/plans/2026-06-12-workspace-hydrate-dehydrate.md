# Workspace and State: Hydrate/Dehydrate Data Path (W4, slice 2)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make Workspace state actually move: a sandbox bound to a Workspace HYDRATES the workspace head revision into its `/workspace` on start, and DEHYDRATES `/workspace` to a new committed WorkspaceRevision on terminate, over the existing content-addressed store (`internal/cas`). This slice adds (1) a bulk workspace transfer primitive to the guest agent (tar a directory out, untar a directory in, over vsock, so a whole `/workspace` moves in one stream instead of per-file round-trips), with host-side hydrate/dehydrate helpers that bridge the CAS (`PutSnapshot`/`Materialize`) to the guest, and (2) the `spec.workspaceRef` binding on SandboxClaim plus the lifecycle orchestration: on activation with a workspaceRef the run path hydrates the head revision; on terminate it dehydrates `/workspace` into a new `WorkspaceRevision` (Committed, lineage `fromClaim`), which the Workspace controller (slice 1) picks up as the new head. The single-writer-per-revision doctrine holds: an active claim is the workspace's writer; terminate commits exactly one revision. Verified: the bulk tar transfer + the CAS round-trip in unit tests and on KVM (a real in-VM `/workspace` hydrated then read back, byte-identical); the binding + the revision lifecycle (claim with workspaceRef -> on terminate a new Committed revision with `fromClaim` lineage, head advances) in envtest with a fake transfer; the real in-VM hydrate/dehydrate on KVM.

**Honesty constraint:** the in-VM data movement (files actually written into and read out of a booted VM's `/workspace`) needs the VMM, so it is proven on real KVM (kvm-test.yaml), the same honest split as the husk in-VM path; on kind/envtest the BINDING + the revision lifecycle are proven object-level with a fake transfer seam, and the docs say so. No claim that data moved that a KVM job does not demonstrate. The dehydrate is the single writer: concurrent active claims on one workspace are rejected or serialized (a workspace bound to an active claim cannot be bound by another; state the rule + enforce it). Content is content-addressed: the dehydrated revision's manifest is a real CAS digest, so an unchanged `/workspace` dedups to the same chunks (a no-op-diff revision is still recorded honestly with the same content digest). Secrets/tokens are never captured into a workspace revision (the dehydrate excludes the secret mount paths; state which paths are excluded). Per-workspace encryption-key pairing (#31) is out of scope here and noted.

**Architecture:** Guest agent (`guest/agent`) gains two vsock ops via the protocol (`internal/vsock`): `TarDir(path) -> stream` (tar a directory to the host) and `UntarDir(path, stream)` (extract a tar into a directory), or a single bidirectional workspace-transfer message; the host streams the tar. Host-side helpers (a new `internal/workspace` package, or in `internal/daemon`): `Hydrate(agent, store, manifestDigest)` = Materialize the manifest to a temp tar / stream + UntarDir into the guest `/workspace`; `Dehydrate(agent, store) -> Manifest` = TarDir `/workspace` from the guest + PutSnapshot into the CAS -> a manifest digest. The SandboxClaim gains `spec.workspaceRef`. The run path (forkd / the claim reconciler): on activation, if workspaceRef set, resolve the Workspace head revision + Hydrate; on terminate, Dehydrate + create a `WorkspaceRevision{workspaceRef, source.fromClaim, contentManifest, phase Committed}`. The Workspace controller (slice 1) advances head on the new committed revision.
