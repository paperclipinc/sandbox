# Husk Pods: Load-Bearing CoW-Across-Memcg Verification (issue #18, PR1)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Execute the FIRST step issue #18 mandates: prove (or disprove) the load-bearing claim of the husk-pods architecture before migrating any controller. The claim: CoW page-cache sharing survives pod memory-cgroup boundaries, so N microVMs restored from ONE snapshot, each placed in its OWN cgroup v2 memory cgroup (as it would be per-pod), still SHARE the snapshot's clean pages (they are not charged N times), while each VM's private DIRTY pages are charged to its own memcg. The issue is explicit: "The design stands or falls here. Write this test and the memcg analysis in docs/husk-pods.md before migrating controllers." This PR builds that test (a real KVM measurement on cgroup v2) and writes the analysis; it does NOT migrate controllers, change the default engine, or claim husk pods are shipped. It is a de-risking investigation with an honest written conclusion.

**Honesty constraint:** this PR makes NO claim that sandboxes are pods or that pod-scoped policy governs them (that is the rest of #18, not this PR). It reports a measured result from real cgroup v2 + Firecracker on the CI runner: the per-process Pss/Rss/Private_Dirty from smaps and the per-memcg memory.current/memory.stat. It states honestly the known cgroup v2 nuance that shared file-backed pages are charged to the FIRST cgroup that faults them (not evenly split), and reconciles that with the CoW-aware metering (#33) which is the honest per-tenant accounting. Every number in docs/husk-pods.md is reproducible from the CI probe phase; no number is invented. If the measurement shows CoW does NOT survive memcg boundaries (e.g. pages re-charged per memcg, or sharing broken), the doc says so plainly and flags that the husk-pods architecture needs rethinking, rather than papering over it.

**Architecture:** A linux-only probe (`cmd/husk-probe`) drives the existing fork engine to restore N sandboxes from one template snapshot, then places each sandbox's Firecracker process into its own cgroup v2 memory cgroup under a probe root (mkdir the memcg dir, write the PID to cgroup.procs), lets them settle, and measures: per-process smaps_rollup (Pss, Rss, Shared_Clean, Private_Dirty) and per-memcg memory.current + memory.stat (file vs anon). It computes the aggregate physical footprint (which, if CoW survives, is roughly one shared snapshot resident set + the sum of per-VM private dirty, far below N x full-VM-size) and the per-memcg dirty attribution, and emits a JSON report + a human summary. The KVM CI runs the probe and asserts the load-bearing properties; docs/husk-pods.md records the model + the measured numbers + the conclusion.

**Context for the implementer:**
- The fork engine restores from a snapshot mem file via mmap (the CoW path); `internal/fork/engine.go` Fork returns a sandbox with a Firecracker PID (`sandbox.Pid`), `readMemoryStats(pid)` already parses /proc/<pid>/smaps_rollup (Private_Clean/Private_Dirty -> unique; the shared accounting). `internal/metering` aggregates CoW-aware (shared-once) totals. The jailer places VMs in cgroup v2 already (`internal/firecracker/jailer.go` --cgroup-version 2); but for THIS probe you may place the Firecracker PID into a probe-created memcg directly (mkdir under /sys/fs/cgroup, echo PID to cgroup.procs) so the test controls the per-VM memcg without the full jailer.
- cgroup v2 memory: a memcg dir has `memory.current` (bytes charged), `memory.stat` (file, anon, etc.). A file-backed page (the snapshot mem file mmap'd MAP_PRIVATE) faulted by a process in memcg A is charged to A; another process in memcg B reading the SAME already-resident page is NOT re-charged. A private write (CoW) allocates an anon page charged to the writer's memcg. To place a PID: write it to `<memcg>/cgroup.procs` (the controller must be enabled in the parent's cgroup.subtree_control). Enabling the memory controller + creating child cgroups needs root + a writable cgroup2 mount (present on the KVM runner).
- The bench/metering CI already builds a template snapshot at `$BENCH_DATA/templates/bench`; reuse it as the snapshot to restore N times.

- [ ] `internal/huskprobe/analyze.go` (pure, unit-tested, no cgroup/KVM): types for a per-VM sample `VMSample{ PID int; Pss, Rss, SharedClean, PrivateDirty, MemcgCurrent, MemcgFile, MemcgAnon int64 }` and a `Report{ VMs []VMSample; AggregatePhysical, NaiveSum, SharedResident, TotalPrivateDirty int64; CoWSurvives bool; ... }`. `Analyze(samples) Report`: AggregatePhysical = an honest estimate of real physical pages = SharedResident (the shared snapshot clean set, counted once, derived from Pss/Rss or memcg file pages) + sum(PrivateDirty); NaiveSum = sum(Rss) (what you would charge if every VM counted its full RSS); the CoW saving = NaiveSum - AggregatePhysical. `CoWSurvives` = AggregatePhysical is materially below NaiveSum AND per-VM PrivateDirty is attributed (each VM has its own dirty charged to its memcg).
